# MPE MaaFramework 项目工作台产品定位与工作区架构

- 状态：已决策
- 决策日期：2026-07-19
- 适用范围：MPE 产品规划、交互设计、Editor、LocalBridge、Desktop 与相关文档

## 1. 文档目的

本文固定 MaaPipelineEditor（MPE）从“Pipeline 编辑器”向“MaaFramework 项目工作台”演进的产品决策，并作为后续需求拆分、架构设计和实现评审的上位约束。

本文规定产品中心、目标用户、顶层工作区及关键边界，不替代具体功能的交互稿、协议设计或技术方案。后续方案可以细化本文，但不得在未更新本决策的情况下与其冲突。

## 2. 背景

MPE 过去以单个 Pipeline 的导入、预览和可视化编辑为中心。受这一定位影响，文件访问策略较为保守，项目中的 ProjectInterface、资源、图片、模型、Agent 代码、运行配置及交付流程没有被统一建模。

真实的 MaaFramework 开发工作并不止于编辑 Pipeline。开发者需要在 Project、ProjectInterface、Bundle、Resource、Pipeline、资源文件、Agent、Controller 和运行结果之间持续切换。继续以单个文件为产品中心，会造成以下问题：

- 项目上下文在不同工具之间反复丢失或重复配置；
- 跨文件引用、资源关系和运行配置无法被完整理解；
- 可视化编辑、代码修改、运行调试和 AI 辅助难以形成闭环；
- 产品能力随功能增加而分散，缺少稳定的信息架构。

因此，MPE 需要从“编辑一种文件”升级为“理解并操作一个 MaaFramework 项目”。

## 3. 核心决策

### 3.1 产品中心

MPE 的产品中心由单个 Pipeline 编辑器调整为单个 MaaFramework 项目工作台。

一个项目会话在任一时刻围绕一个 MaaFramework 项目建立完整上下文。多窗口、最近项目和项目切换可以由宿主能力提供，但不改变“一项目会话一项目”的领域模型。

Pipeline 可视化编辑继续作为 MPE 的核心优势，但其身份调整为 Elaborator 中的专业编辑器之一，不再单独决定整个产品的数据模型和信息架构。

### 3.2 产品定义

> MPE 是面向 MaaFramework 项目开发者与维护者的一体化项目工作台，覆盖项目理解、开发调试、运行验收、交付维护与智能协作。

### 3.3 核心用户

MPE 首要服务于 MaaFramework 项目开发者与维护者，包括项目创建者、资源开发者、Agent 开发者、调试与发布维护人员。

临时查看者、学习者和项目最终用户仍可从部分能力中受益，但不作为顶层产品结构的首要决策对象。

### 3.4 顶层工作区

MPE 在项目工作台内建立三个顶层功能工作区：

| 中文名 | 英文名 | 核心目标 |
| --- | --- | --- |
| 编辑器 | Elaborator | 理解、创建、修改、验证和打磨项目 |
| 运行器 | Embodier | 将项目文件转化为可操作、可感知的真实产品 |
| 智能体 | Erranter | 以项目智能体方式探索、修改、验证和运行项目 |

三个功能工作区的正式展示名为“编辑器 - Elaborator”“运行器 - Embodier”“智能体 - Erranter”。实现界面中，中文名与英文名必须成对出现；排版可以使用空格、换行或分隔符，但不得只显示其中一种语言，也不得改变对应关系。

“编辑器”“运行器”“智能体”是产品工作区，不是部署方式。纯 Web、Web + LocalBridge 和 Desktop 属于部署形态与能力层级，两套概念不得混用。

三组名称共同表达 MPE 对项目全生命周期的角色分工：

- **Elaborator**：不厌其烦地打磨项目中的每一处细节；
- **Embodier**：成为代码的真实化身，使项目从文件转变为可操作、可感知的产品；
- **Erranter**：以具有探索精神的游侠姿态理解项目并自主推进目标。

### 3.5 术语约定

| 术语 | 含义 |
| --- | --- |
| 项目工作台 | MPE 的整体产品形态，提供项目级开发、运行和 AI 能力 |
| 项目会话 | 一次已打开项目的完整上下文；一个项目会话只对应一个项目 |
| 功能工作区 | 项目工作台内的顶层功能区域，即编辑器 Elaborator、运行器 Embodier、智能体 Erranter |
| 部署形态 | 纯 Web、Web + LocalBridge、Desktop，决定能力由何种宿主提供 |
| MaaFramework Agent | MaaFramework 项目中的运行时扩展机制，不是 MPE 智能体 Erranter |

## 4. 产品北极星

MPE 的长期验收标准是：

> 打开一个 MaaFramework 项目后，开发者无需离开 MPE，就能理解它、修改它、验证它、运行它、定位问题并完成交付。

完整生命周期至少包含：

1. 打开或创建项目，识别其结构与能力；
2. 阅读 Pipeline、ProjectInterface、资源和 Agent 之间的关系；
3. 使用可视化或代码方式修改项目；
4. 执行静态验证、资源检查和运行调试；
5. 以接近真实应用的方式配置并运行任务；
6. 根据日志、截图、事件与运行轨迹定位问题；
7. 完成构建、打包、本地化、版本检查等交付维护工作；
8. 让 AI 在受控权限下参与上述流程。

## 5. 统一项目上下文与状态权威

三个工作区必须建立在同一个项目上下文之上，而不是形成三套独立应用或数据模型。

统一项目上下文至少包括：

- 项目根目录、项目身份与信任状态；
- ProjectInterface 及其 import、task、option、preset、group 等结构；
- Resource、Bundle、Pipeline、图片、模型与 Agent 的语义关系；
- 当前文件、当前节点、当前任务等导航和选择状态；
- 文件修改、未保存状态、外部变更、差异与撤销记录；
- 项目级诊断、引用关系、搜索结果与构建结果；
- Controller、设备、运行会话、日志、截图和调试产物；
- AI 会话、工具权限、操作记录和检查点。

统一上下文不表示所有状态都写入项目目录。不同状态必须有明确且唯一的权威来源：

| 状态类别 | 权威来源 | 持久化与恢复原则 |
| --- | --- | --- |
| 项目身份、当前项目与能力状态 | 项目会话服务 | 绑定规范化后的项目位置、存储适配器和能力清单；切换项目必须创建或恢复对应项目会话 |
| 已保存的项目内容 | 当前项目存储适配器；本地完整模式下通常为磁盘文件 | 不复制为 MPE 私有项目格式；重新打开后从源文件恢复 |
| 当前文件、节点、任务及导航历史 | Editor 中唯一的项目会话状态存储 | 三个功能工作区只读取或更新同一份会话状态；是否跨启动恢复另行设计 |
| 打开文档、未保存内容、撤销记录与外部变更冲突 | 统一文档服务中的当前文档模型 | 保存前不得被磁盘监听结果静默覆盖；冲突解决和会话恢复策略另行设计 |
| 语义图、搜索索引和诊断缓存 | 由项目内容派生的项目语义服务 | 必须可以丢弃并从项目内容重建 |
| Controller 与运行会话 | LocalBridge 中的实际运行对象 | 进程或连接失效后按真实状态报告，不伪造恢复 |
| 日志、截图、追踪等运行产物 | 运行 artifact 存储 | 与运行会话关联；只有显式操作才写入项目目录 |
| 构建、测试和项目命令状态 | LocalBridge 作业执行服务；纯 Web 仅限浏览器安全作业 | 结果写入 artifact 与审计存储；只有项目命令明确产生的文件才进入项目内容 |
| 项目信任、用户设置和授权记录 | 受控的应用配置与审计存储 | 不默认随项目文件传播；支持撤销，具体周期另行设计 |
| AI 会话、计划和检查点 | AI 会话存储 | 与项目身份关联；通过工具调用标识引用独立的副作用审计，导出到项目必须由用户显式触发 |

典型闭环应当能够成立：

```text
编辑器修改 -> 静态验证 -> 运行器验收 -> 发现异常 -> 跳回源码或节点
    ^                                             |
    +--------- 智能体分析、提出修改并受控执行 <---+
```

项目存储适配器中的文件是项目内容的事实来源；具备完整本地文件能力时，该适配器直接对应项目磁盘文件。MPE 不应默认将项目复制进私有格式后形成第二份项目内容来源。

## 6. 三个工作区的职责

### 6.1 编辑器：Elaborator

Elaborator 负责项目的理解、创作与验证，整合现有可视化视图和类 IDE 的代码、文件与资源查看能力。

主要职责包括：

- 项目文件树、标签页、搜索、跳转和历史导航；
- Pipeline 可视化编辑与源码编辑之间的一致映射；
- `interface.json`、Pipeline JSON/JSONC、配置和 Agent 代码编辑；
- 图片、模型、日志及其他项目资源的合适预览；
- 跨文件引用、重命名、缺失资源和协议错误诊断；
- 截图、ROI、取色、OCR、模板匹配等开发工具；
- 调试入口、运行结果回链和交付维护工具。

Elaborator 不以复刻完整通用 IDE 为目标。其差异化价值是 MaaFramework 语义理解、可视化与代码协同，以及从编辑到运行反馈的短路径。

编辑 MaaFramework 文件时，应尽量使用精确位置修改，保留 JSONC 注释、原有格式和字段顺序。不得默认通过普通反序列化后整体重写源文件。

### 6.2 运行器：Embodier

Embodier 是工作台内的通用运行器。它依据项目真实声明组织 Controller、Resource、Task、Option、Preset 等内容，并通过真实 MaaFramework 运行时执行项目。

主要职责包括：

- 解析并呈现 ProjectInterface 定义的任务和配置；
- 选择、配置和连接 Controller 或设备；
- 加载 Resource，创建并管理真实运行会话；
- 启动、停止和观察任务执行；
- 展示日志、事件、截图、运行轨迹、结果与错误；
- 从运行异常直接定位至编辑器 Elaborator 中的文件、节点或配置；
- 提供接近项目最终应用的验收和日常操作体验。

`Embodier` 意为“代码的真实化身”：它让静态的项目文件通过真实 MaaFramework 运行时成为可操作、可观察、可感知的产品。它不是模拟器；凡可使用真实 MaaFramework 行为的场景，均应以真实运行结果为准，不自行推测框架语义。

运行器首先服务项目开发、验收和维护。它可以承载实际项目运行，但现阶段不以替代所有面向最终用户的独立通用客户端为前提。

### 6.3 智能体：Erranter

Erranter 是项目级智能体。它应理解统一项目上下文，并调用与 Elaborator、Embodier 共同依赖的文件、语义、验证和运行服务完成目标，不得依赖或自动操作另两个工作区的 UI 内部实现。

主要职责包括：

- 解释项目结构、Pipeline 逻辑与跨文件关系；
- 搜索问题、制定计划并提出可审查的修改；
- 在授权后编辑文件、生成资源或调整配置；
- 调用静态验证、构建、测试和运行能力；
- 观察日志、截图与运行轨迹，基于反馈继续迭代；
- 支持探索型、自主型游戏任务等高阶场景。

“自主游戏智能体”是 Erranter 的重要应用方向，但不是其唯一产品定义。Erranter 首先是能够参与项目全生命周期的项目智能体。

Erranter 与 MaaFramework Agent 必须在术语和实现上严格区分：前者是 MPE 的智能体，后者是 MaaFramework 项目的运行时扩展机制。

AI 权限至少应区分：

1. 读取项目和运行信息；
2. 提出修改但不写入；
3. 写入项目文件；
4. 启动进程、构建或任务；
5. 连接和控制真实设备。

所有写入和有副作用的操作都应具备明确授权、差异展示、检查点、撤销或恢复方案及可追溯记录。AI 不得绕过项目信任和运行安全边界。

## 7. 文件访问与项目信任

原有保守文件策略不再作为产品能力上限，调整为按风险分层：

- 浏览层：允许查看项目范围内的广泛文件类型，并为未知格式提供安全的元信息或文本查看；
- 编辑层：允许编辑适合文本处理的文件，对已知 MaaFramework 格式提供语义能力；
- 执行层：Agent、脚本、构建命令、外部程序和设备控制必须经过项目信任及显式授权；
- 越界层：项目根目录之外的访问、符号链接、路径穿越和敏感文件读取必须受明确策略限制。

“能够查看”不等于“能够执行”，“能够编辑”也不等于“自动运行”。打开项目不得自动启动 Agent、脚本、构建命令或设备操作。

项目信任和单次操作授权是不同概念。信任项目不等于一次性授权所有 AI、命令、网络和设备操作。安全设计至少必须覆盖：

- 默认拒绝项目边界外访问，并在解析符号链接后再次校验真实路径；
- 分别授权读取、写入、创建、删除、重命名、进程启动、网络访问和设备控制；
- 对命令、参数、工作目录、环境变量及可继承密钥设置明确范围；
- 区分单次、当前会话和持久授权，并提供查看与撤销入口；
- 对发送给外部 AI 服务的项目内容、日志、截图、密钥和其他敏感数据执行数据外发策略；
- 将有副作用的操作及其授权依据写入审计记录。

敏感文件范围、授权交互和持久化格式由专项安全设计确定；在其完成前，高风险能力应按默认拒绝处理。

## 8. 架构约束

### 8.1 Editor

Editor 负责工作区界面、统一的项目会话状态和能力编排。当前文件、节点、任务及导航历史只能存在于每个项目会话唯一的共享状态存储中，不得由三个功能工作区分别维护。Editor 通过统一的项目服务访问文件、语义和运行能力，不在不同工作区重复实现项目解析和运行逻辑。

纯 Web 环境可以提供受能力限制的项目体验，但必须诚实反映文件系统、原生运行时和设备能力是否可用，不得以静默失败或伪运行代替真实能力。

### 8.2 LocalBridge

LocalBridge 继续作为唯一业务后端，并从 Pipeline 辅助后端演进为项目与运行能力后端，主要承载：

- 受项目根目录约束的文件访问、监听、搜索和变更协调；
- MaaFramework 项目结构识别、语义索引和项目级诊断；
- Controller、Resource、Tasker、AgentClient 及运行会话编排；
- 日志、截图、事件、追踪和其他 artifact 管理；
- 项目 Agent、构建命令及其他授权子进程的执行与生命周期管理；
- AI 可调用本地工具的受控执行、最终权限校验和副作用审计。

协议应以能力清单和稳定领域对象为边界，避免将前端组件状态直接固化为后端协议。

### 8.3 Desktop

Desktop 继续承担桌面应用宿主职责，包括 LocalBridge 进程生命周期、原生目录选择、系统集成、私有运行环境和更新。项目自身的 Agent、构建命令和其他子进程由 LocalBridge 作业执行服务管理，不由 Desktop 直接执行。不得在 Rust 层复制 LocalBridge 的项目语义与 MaaFramework 业务逻辑。

Desktop 宿主能力默认只接受用户直接操作或受控的应用流程，不直接暴露为 Erranter 工具。未来如需让 AI 调用目录、系统集成、更新等具有副作用的宿主能力，必须先为 Desktop 建立与 LocalBridge 等价的授权、范围、撤销和权威审计契约，并形成单独设计决策。

### 8.4 共享能力与所有权

功能工作区负责呈现和编排，不拥有只能藏在 UI 内部的项目或运行核心能力。逻辑所有权与具体运行进程可以不同，但所有消费者必须遵循同一领域契约。

| 共享能力 | 逻辑所有者 | 主要呈现位置 | 其他消费者 |
| --- | --- | --- | --- |
| 项目存储、文件修改与变更协调 | 统一项目服务 | 编辑器 Elaborator | 运行器 Embodier、智能体 Erranter |
| 项目结构、语义图、搜索与诊断 | 项目语义服务 | 编辑器 Elaborator | 运行器 Embodier、智能体 Erranter |
| Controller、Resource、Tasker 与运行会话 | LocalBridge 运行服务 | 运行器 Embodier | 编辑器 Elaborator 的调试入口、智能体 Erranter |
| 日志、截图、事件、追踪与 Replay | LocalBridge artifact/运行服务 | 运行器 Embodier、编辑器 Elaborator | 智能体 Erranter |
| AI 会话、计划、工具调度与授权请求 | AI 编排服务 | 智能体 Erranter | 编辑器 Elaborator、运行器 Embodier 的 AI 入口 |
| 本地能力授权、最终强制执行与副作用审计 | LocalBridge 授权层 | 授权界面与审计视图 | 所有功能工作区和 AI 编排服务 |
| Agent、构建命令和其他项目子进程 | LocalBridge 作业执行服务 | 编辑器 Elaborator、运行器 Embodier | 智能体 Erranter |
| LocalBridge 生命周期、原生目录与系统集成 | Desktop 宿主 | 项目工作台外壳 | 仅由用户或受控应用流程使用，默认不向 Erranter 暴露 |

上述能力应先建设为可复用服务，再由三个工作区组合使用。不得为某个工作区建立无法被其他工作区复用的平行实现，除非其领域边界确实不同。

授权界面和 AI 编排服务只能发起授权请求，不能作为本地副作用的最终安全边界。LocalBridge 是项目文件、进程、网络和设备能力的最终强制执行点，并保存其权威审计；纯 Web 中则由提供对应浏览器安全能力的适配器强制执行。Desktop 宿主能力不得借由 Editor 或 AI 绕过这一原则：默认不向 AI 暴露，未来开放时由 Desktop 自身作为最终能力提供者执行等价鉴权与权威审计。拒绝或撤销授权后，能力提供者必须拒绝后续调用，即使 UI 或 AI 会话仍保留旧状态。

### 8.5 部署能力基线

三个功能工作区在所有部署形态中保持相同概念，但可用工具按真实能力降级：

| 能力 | 纯 Web | Web + LocalBridge | Desktop |
| --- | --- | --- | --- |
| 项目承载 | 受浏览器项目存储适配器限制 | LocalBridge 根目录内的本地项目 | 原生目录选择后交由 LocalBridge |
| 浏览、编辑与可视化 | 可用，受浏览器文件能力限制 | 完整本地能力 | 完整本地能力 |
| 文件监听、全项目搜索与增强诊断 | 按浏览器能力降级 | 可用 | 可用 |
| MaaFramework 原生运行时与设备 | 不可用 | 可用 | 可用 |
| Agent、构建和本地进程 | 不可用 | 授权后可用 | 授权后可用 |
| Erranter 工具 | 仅浏览器安全工具 | 按权限调用 LocalBridge 工具 | 按权限调用 LocalBridge 工具，不直接调用 Desktop 宿主能力 |

纯 Web 项目存储适配器的具体形态仍待专项设计。任何不可用能力都必须通过能力清单显式反馈，不得显示为可执行后再静默失败。

## 9. 现有能力的归属

| 现有能力 | 新定位中的归属 |
| --- | --- |
| Pipeline 画布、字段编辑、编译器与解析器 | 编辑器 Elaborator 的 Pipeline 专业编辑器 |
| 本地文件扫描、保存、监听与跨文件搜索 | 统一项目服务 |
| 图片预览、截图、OCR、ROI、取色等工具 | 编辑器 Elaborator，共享 Controller 与资源上下文 |
| 流程级调试、事件续传、Replay 和日志 | 统一运行服务，由编辑器 Elaborator 与运行器 Embodier 分别呈现 |
| 节点搜索、AI 补全和探索模式 | 智能体 Erranter 的早期能力，逐步迁移到项目级智能体工具 |
| LocalBridge | 项目、运行和 AI 工具的唯一业务后端 |
| Desktop | 工作台的桌面宿主，不复制业务逻辑 |

已有能力应尽可能提升和复用，而不是因工作区重组而重新实现。

## 10. 演进顺序

后续落地按依赖关系优先推进：

1. **项目底座**：项目打开与识别、统一项目上下文、文件服务、信任模型、语义索引和能力清单；
2. **编辑器 Elaborator 整合**：将现有画布纳入项目工作区，补充代码、文件、资源查看及跨文件诊断；
3. **运行器 Embodier 建设**：ProjectInterface 驱动界面、真实运行会话、设备配置、日志和运行回链；
4. **智能体 Erranter 建设**：基于前述稳定的文件、验证、最小构建/测试和运行接口，实现项目理解、编辑、验证、运行和反馈迭代；
5. **交付闭环**：在最小构建/测试接口之上完善打包、本地化、版本检查、发布维护及跨工作区体验。

该顺序表达架构依赖，不要求每一阶段完全结束后才能开始下一阶段。可交付功能应尽早形成纵向闭环，但不得绕过统一项目底座制造临时平行模型。

## 11. 非目标

当前决策不要求：

- 将 MPE 建设为适用于任意软件项目的通用 IDE；
- 同时维护三套互不相通的工作区状态和后端实现；
- 通过模拟逻辑替代真实 MaaFramework 运行时；
- 让 AI 获得默认无限制的文件、进程或设备权限；
- 立即替代所有 MaaFramework 最终用户通用客户端；
- 以兼容旧导入模型为由长期保留重复的数据源和废弃架构；
- 让多项目管理需求破坏“一项目会话一项目”的核心模型。

单文件粘贴、临时审阅等轻量入口可以保留，但应被视为项目工作台的降级或临时使用方式，不再反向决定核心架构。

## 12. 成功标准

当以下条件成立时，可以认为项目工作台定位已经形成有效落地：

- MPE 能识别标准 MaaFramework 项目的核心结构和关联关系；
- 用户可在项目范围内浏览、编辑、搜索和验证文件；
- 可视化与代码修改共享同一事实来源，且不会无故破坏 JSONC 注释和格式；
- 编辑器 Elaborator、运行器 Embodier 和智能体 Erranter 切换时保留同一项目上下文；
- 运行器 Embodier 使用真实项目配置和 MaaFramework 运行时；
- 运行错误、日志、截图和轨迹能够回链到开发上下文；
- AI 复用正式项目和运行工具，所有副作用均可授权、审查和追溯；
- 不同部署形态能够按能力降级，并清楚告知不可用能力；
- 开发者可以在 MPE 内完成从项目理解到交付维护的主要闭环。

首个完整版本至少应通过以下端到端场景：

1. 打开包含 ProjectInterface import、多个 Resource、JSONC Pipeline、图片和 Agent 的标准测试项目，能够建立关系图并报告故意设置的缺失引用；
2. 分别从画布和源码修改同一 Pipeline，保存后语义一致，修改范围外的 JSONC 注释、字段顺序和格式不被重写；
3. 文件在外部变化且本地存在未保存编辑时，明确提示冲突，不静默覆盖任一方；
4. 在 Web + LocalBridge 与 Desktop 中运行 ProjectInterface 任务，收集日志和产物，并在回调信息足够时定位到对应任务、文件或节点；
5. 在纯 Web 中打开同一项目时，开发能力按存储适配器工作，原生运行和本地进程能力明确显示不可用；
6. 在编辑器 Elaborator、运行器 Embodier 和智能体 Erranter 之间切换，项目身份、相关选择、诊断和运行会话不需要重新导入或重复配置；
7. Erranter 在只读权限下无法写文件；获得写入权限后先展示差异；启动进程和控制设备需要独立授权，并留下可查询记录；
8. 面对无法识别或部分损坏的项目时，保留安全浏览能力并给出可操作诊断，不静默套用错误的项目模型；
9. 通过共享作业服务完成至少一次测试、构建和打包，产物来自当前项目内容，成功或失败结果均可追溯，且阶段 4 的最小接口与阶段 5 的完整交付体验使用同一服务；
10. 使用安全测试项目尝试路径穿越、指向项目外的符号链接、未授权删除或重命名、进程启动、网络访问、AI 数据外发和设备控制；默认均被最终能力提供者拒绝，授权后只能在批准范围和期限内执行，撤销后下一次调用立即失败且审计记录仍可查询；当前版本中，AI 直接调用 Desktop 宿主能力必须始终被拒绝，只有未来完成专项决策并实现等价安全契约后，才增加相应的授权验收。

## 13. 后续方案评审清单

涉及产品结构、文件、运行或 AI 的需求，在进入实现前应回答：

1. 该能力属于哪个顶层工作区，是否也应被其他工作区复用？
2. 它操作的是哪个 MaaFramework 项目实体或运行时对象？
3. 它如何接入统一项目上下文，事实来源是什么？
4. Editor、LocalBridge 和 Desktop 的职责分别是什么？
5. 纯 Web、Web + LocalBridge 和 Desktop 中如何声明或降级能力？
6. 是否涉及外部文件、进程、Agent、密钥或真实设备，授权边界是什么？
7. 修改是否精确、可撤销、可审查，并能处理外部文件变化？
8. 运行结果能否回到对应文件、节点、任务或配置？
9. 是否复用了现有实现，或有充分理由建立新模块？
10. 需要补充哪些前后端测试、用户文档和迁移说明？

## 14. 待专项设计事项

以下问题需要在对应功能设计中继续决策，但不影响本文确立的方向：

- MaaFramework 项目的精确识别规则及非标准目录结构处理；
- 单根目录、外部 import、多资源根目录与项目边界之间的关系；
- 纯 Web 中项目目录、压缩包或临时项目的承载方式；
- 项目信任的持久化、撤销和权限粒度；
- 多 Controller、多运行会话和并发任务的交互模型；
- ProjectInterface 对运行界面的覆盖范围、扩展机制和具体信息架构；
- AI 模型、上下文索引、工具协议和长期会话存储方案；
- AI 数据外发、网络访问、环境变量与敏感信息处理策略；
- 构建、打包、发布与第三方项目脚手架之间的职责边界；
- 精确文本编辑在不同 JSON/JSONC 风格和异常输入下的保真验证；
- 产品名称、README、官网和用户文档的品牌迁移节奏。

这些事项应分别形成设计文档或决策记录，并引用本文作为上位依据。
