# MPE MaaFramework 项目工作台产品定位与工作区架构

- 状态：已决策
- 决策日期：2026-07-21
- 适用范围：MPE 产品规划、交互设计、Editor、LocalBridge、Desktop 与相关文档

## 1. 文档目的

本文固定 MaaPipelineEditor（MPE）从“Pipeline 编辑器”向“MaaFramework 项目工作台”演进的产品决策，并作为后续需求拆分、架构设计和实现评审的上位约束。

MPE 的产品中心是一个 MaaFramework 项目。Pipeline 可视化编辑仍是核心优势，但必须建立在统一项目上下文、统一文档模型和真实 MaaFramework 运行时之上。

配套实施方案：[项目文件操作与 Pipeline 双视图整改方案](./项目文件操作与Pipeline双视图整改方案.md)。该文档细化统一文件操作、存储适配器、Pipeline 画布/源码双视图及分阶段迁移要求。

## 2. 产品定位

> MPE 是面向 MaaFramework 项目开发者与维护者的一体化项目工作台，覆盖项目理解、开发调试、运行验收与交付维护。

首要用户包括项目创建者、资源开发者、Agent 开发者、调试人员和发布维护人员。临时查看者和学习者可以使用部分能力，但不作为顶层产品结构的首要决策对象。

一个项目会话在任一时刻只围绕一个 MaaFramework 项目建立完整上下文。多窗口、最近项目和项目切换属于宿主能力，不改变“一项目会话一项目”的领域模型。

## 3. 顶层工作区

MPE 项目工作台包含两个顶层功能工作区：

| 中文名 | 英文名 | 核心目标 |
| --- | --- | --- |
| 编辑器 | Elaborator | 理解、创建、修改、验证和打磨项目 |
| 运行器 | Embodier | 将项目文件转化为可操作、可感知的真实产品 |

正式展示名为“编辑器 - Elaborator”和“运行器 - Embodier”。中文名与英文名必须成对出现。

功能工作区与部署形态是两套概念。纯 Web、Web + LocalBridge 和 Desktop 属于部署形态，不得与工作区混用。

MaaFramework Agent 是项目中的运行时扩展机制，不是 MPE 工作区。它作为项目文件、运行配置和子进程被编辑器与运行器共同管理。

## 4. 产品北极星

打开一个 MaaFramework 项目后，开发者无需离开 MPE，就能完成以下主要流程：

1. 打开或创建项目，识别其结构与能力；
2. 阅读 Pipeline、ProjectInterface、资源和 Agent 之间的关系；
3. 使用可视化或代码方式修改项目；
4. 执行静态验证、资源检查和运行调试；
5. 按真实项目配置运行任务；
6. 根据日志、截图、事件与运行轨迹定位问题；
7. 完成构建、打包、本地化、版本检查等交付维护工作。

## 5. 统一项目上下文与状态权威

两个工作区必须建立在同一项目上下文之上。统一上下文至少包括：

- 项目根目录、项目身份与信任状态；
- ProjectInterface 及其 import、task、option、preset、group 等结构；
- Resource、Bundle、Pipeline、图片、模型与 Agent 的语义关系；
- 当前文件、节点、任务及导航选择；
- 文件修改、未保存状态、外部变更、差异与撤销记录；
- 项目级诊断、引用关系、搜索结果与构建结果；
- Controller、设备、运行会话、日志、截图和调试产物。

不同状态必须有明确且唯一的权威来源：

| 状态类别 | 权威来源 | 持久化与恢复原则 |
| --- | --- | --- |
| 项目身份、当前项目与能力状态 | 项目会话服务 | 绑定规范化项目位置、存储适配器和能力清单 |
| 已保存项目内容 | 当前项目存储适配器 | 不复制为 MPE 私有项目格式，重新打开后从源文件恢复 |
| 当前文件、节点、任务及导航历史 | Editor 项目会话状态 | 两个工作区读写同一份会话状态 |
| 未保存内容、撤销记录与外部变更冲突 | 统一文档服务 | 保存前不得被磁盘监听结果静默覆盖 |
| 语义图、搜索索引和诊断缓存 | 项目语义服务 | 可丢弃并从项目内容重建 |
| Controller 与运行会话 | LocalBridge 运行服务 | 连接失效后按真实状态报告，不伪造恢复 |
| 日志、截图、追踪等产物 | 运行 artifact 存储 | 与运行会话关联，显式操作才写入项目目录 |
| 构建、测试和项目命令状态 | LocalBridge 作业执行服务 | 结果写入 artifact 与审计存储 |
| 用户设置与授权记录 | 受控应用配置与审计存储 | 不默认随项目文件传播，支持撤销 |

项目存储适配器中的文件是项目内容的事实来源。具备完整本地文件能力时，适配器直接对应项目磁盘文件。

## 6. 工作区职责

### 6.1 编辑器：Elaborator

Elaborator 负责项目理解、创作与验证，主要包括：

- 项目文件树、标签页、搜索、跳转和历史导航；
- Pipeline 可视化编辑与源代码编辑之间的一致映射；
- `interface.json`、Pipeline JSON/JSONC、配置和 Agent 代码编辑；
- 图片、模型、日志及其他项目资源预览；
- 跨文件引用、重命名、缺失资源和协议错误诊断；
- 截图、ROI、取色、OCR、模板匹配等开发工具；
- 调试入口、运行结果回链和交付维护工具。

编辑 MaaFramework 文件时应优先使用精确位置修改，保留 JSONC 注释、原有格式和字段顺序，不得默认整体反序列化后重写。

### 6.2 运行器：Embodier

Embodier 是工作台内的通用运行器，依据项目真实声明组织 Controller、Resource、Task、Option 和 Preset，并通过真实 MaaFramework 运行时执行项目。

主要职责包括：

- 解析并呈现 ProjectInterface 定义的任务和配置；
- 选择、配置和连接 Controller 或设备；
- 加载 Resource，创建并管理真实运行会话；
- 启动、停止和观察任务执行；
- 展示日志、事件、截图、运行轨迹、结果与错误；
- 从运行异常定位至编辑器中的文件、节点或配置；
- 提供接近项目最终应用的验收体验。

运行器不使用模拟逻辑代替可用的真实 MaaFramework 行为。

## 7. 文件访问与项目信任

文件能力按风险分层：

- 浏览层：查看项目范围内的广泛文件类型；
- 编辑层：编辑适合文本处理的文件，并为已知 MaaFramework 格式提供语义能力；
- 执行层：Agent、脚本、构建命令和外部程序必须经过项目信任及显式授权；
- 越界层：项目根目录之外的访问、符号链接和敏感文件读取必须受策略限制。

打开项目不得自动启动 Agent、脚本、构建命令或设备操作。安全设计至少覆盖：

- 默认拒绝项目边界外访问，并在解析符号链接后重新校验真实路径；
- 分别授权读取、写入、创建、删除、重命名、进程启动、网络访问和设备控制；
- 对命令、参数、工作目录、环境变量及可继承密钥设置明确范围；
- 区分单次、当前会话和持久授权，并提供查看与撤销入口；
- 将有副作用的操作及授权依据写入审计记录。

## 8. 架构约束

### 8.1 Editor

Editor 负责工作区界面、统一项目会话状态和能力编排。当前文件、节点、任务和导航历史只能存在于每个项目会话唯一的共享状态中。

Editor 通过稳定服务接口访问文件、语义、验证、运行和模型 API 底座，不得通过自动操作其他工作区 UI 来复用能力。

### 8.2 LocalBridge

LocalBridge 是本地完整模式下唯一的业务后端和本地权限边界，负责：

- 项目扫描、文件读写、监听和资源访问；
- MaaFramework Controller、Resource、Tasker 与运行会话；
- 日志、截图、事件、追踪和 artifact 管理；
- Agent、构建命令及其他授权子进程的生命周期；
- 模型 API 的普通请求与流式代理。

协议以能力清单和稳定领域对象为边界，避免固化前端组件状态。

### 8.3 Desktop

Desktop 只承担桌面宿主职责，包括 LocalBridge 生命周期、原生目录选择、系统集成、私有运行环境和更新。项目语义与 MaaFramework 业务逻辑不得复制到 Rust 层。

### 8.4 共享能力与所有权

| 共享能力 | 逻辑所有者 | 主要呈现位置 | 其他消费者 |
| --- | --- | --- | --- |
| 项目存储、文件修改与变更协调 | 统一项目服务 | 编辑器 | 运行器 |
| 项目结构、语义图、搜索与诊断 | 项目语义服务 | 编辑器 | 运行器 |
| Controller、Resource、Tasker 与运行会话 | LocalBridge 运行服务 | 运行器 | 编辑器调试入口 |
| 日志、截图、事件、追踪与 Replay | LocalBridge artifact/运行服务 | 运行器、编辑器 | 共享调试视图 |
| 本地能力授权与副作用审计 | LocalBridge 授权层 | 授权界面与审计视图 | 所有工作区 |
| Agent、构建命令和项目子进程 | LocalBridge 作业执行服务 | 编辑器、运行器 | 交付流程 |
| 模型 API 请求、Provider 适配与代理 | 模型 API 底座 | 设置面板 | 未来显式接入的调用方 |
| LocalBridge 生命周期与系统集成 | Desktop 宿主 | 项目工作台外壳 | 用户直接操作 |

## 9. 部署能力基线

| 能力 | 纯 Web | Web + LocalBridge | Desktop |
| --- | --- | --- | --- |
| 项目浏览与编辑 | 浏览器安全存储能力 | 完整本地项目 | 完整本地项目 |
| MaaFramework 原生运行时与设备 | 不可用 | 可用 | 可用 |
| Agent、构建和本地进程 | 不可用 | 授权后可用 | 授权后可用 |
| 模型 API 直连 | 服务支持 CORS 时可用 | 可用 | 可用 |
| 模型 API 本地代理 | 不可用 | 可用 | 可用 |
| 原生目录选择与系统集成 | 不可用 | 不可用 | 可用 |

不可用能力必须通过能力清单显式反馈，不得显示为可执行后再静默失败。

## 10. 演进顺序

1. 建立统一项目会话、文档服务、存储适配器和能力清单；
2. 将现有画布纳入 Elaborator，补充代码、文件、资源和跨文件诊断；
3. 建设 Embodier 的 ProjectInterface 驱动界面、真实运行会话、设备配置和运行回链；
4. 在共享作业服务上完善构建、测试、打包、本地化、版本检查与发布维护。

现有可复用能力应提升为共享服务，不得为不同工作区建立平行的数据模型或后端实现。

## 11. 非目标

- 同时维护多套互不相通的工作区状态和后端实现；
- 通过模拟逻辑替代真实 MaaFramework 运行时；
- 立即替代所有 MaaFramework 最终用户通用客户端；
- 以兼容旧导入模型为由长期保留重复数据源和废弃架构；
- 让多项目管理破坏“一项目会话一项目”的核心模型。

## 12. 验收标准

- 用户可在项目范围内浏览、编辑、搜索和验证文件；
- 可视化与代码修改共享同一事实来源，不破坏 JSONC 注释和格式；
- 编辑器与运行器切换时保留同一项目上下文；
- 运行器使用真实项目配置和 MaaFramework 运行时；
- 运行错误、日志、截图和轨迹能够回链到开发上下文；
- 不同部署形态按能力降级并明确告知不可用能力；
- 路径穿越、项目外符号链接、未授权文件操作、进程启动和设备控制均由最终能力提供者拒绝；
- 模型 API 底座不保存会话、不注入预设消息，仅处理调用方显式提供的请求。

## 13. 后续方案评审清单

1. 能力属于哪个工作区，是否需要共享？
2. 它操作哪个 MaaFramework 项目实体或运行时对象？
3. 事实来源和状态权威在哪里？
4. 纯 Web、Web + LocalBridge、Desktop 分别如何实现或降级？
5. 是否涉及未保存内容、外部变更或多视图一致性？
6. 是否涉及文件、进程、网络、密钥或设备权限？
7. 是否可复用现有能力，废弃实现是否已直接移除？
8. 是否具备静态检查、运行反馈和可追溯验证？
