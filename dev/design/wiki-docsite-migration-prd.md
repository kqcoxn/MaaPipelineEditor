# 文档站迁移至 Wiki 系统 PRD

状态：P0 Frozen  
日期：2026-05-03  
范围：基于当前 `docsite/docs` 与现有内置 Wiki 能力，规划 MPE 文档从外部文档站向应用内 Wiki 的内容迁移方案。本 PRD 仅定义产品与内容规划，不包含本轮实现代码。

## 1. Executive Summary

### Problem Statement

当前 MPE 的主要使用文档位于外部 VitePress 文档站，适合完整阅读与对外分发，但不适合用户在编辑、调试、工具箱、AI 辅助等高频交互场景下就地查询。仓内已经存在内置 Wiki 阅读器、搜索、分享链接与上下文触发能力，但当前只覆盖少量 `debug` / `toolbox` 示例条目，无法承接整体现有使用文档。

同时，现有 docsite 与内置 Wiki 的内容模型并不一致：docsite 以 Markdown 页面组织，Wiki 则以 `Entry -> Module -> Step -> Block` 结构组织，并且搜索索引、分享目标、上下文入口都需要显式配置。如果直接一比一搬运页面，会得到一套结构不适配、维护成本更高、阅读体验更差的新文档。

### Proposed Solution

本次迁移采用“按内容属性分流”而非“按页面逐个平移”的策略：

1. 保留 docsite 承载安装、部署、协议、开发协作、长篇参考等外部文档。
2. 优先把高频、任务型、上下文相关的操作帮助迁入内置 Wiki。
3. Wiki 的信息架构沿用现有 `Entry / Module / Step` 模型，围绕 `开始使用`、`工作流编辑`、`工具箱`、`调试`、`本地能力`、`AI 辅助`、`迁移` 七个主题域建设。
4. 通过全局入口、搜索、哈希分享与界面上下文触发器，形成“在操作处看到对应文档”的闭环。

### Success Criteria

- 完成对现有 27 个 guide 页面（含首页与目录页）的迁移分流清单，明确 `Wiki First`、`Hybrid`、`Docsite First`、`Portal` 四类归属。
- 首批 Wiki 覆盖编辑器内 80% 以上高频操作面：节点、字段、连接、导入导出、工具箱、调试、基础迁移。
- 所有迁入 Wiki 的模块都具备 `title`、`summary`、`keywords`、`searchText` 与哈希分享目标，且能从全局入口或上下文入口到达。
- 首批上线后，用户搜索“ROI”“从节点运行”“导入旧 Pipeline”“字段面板”等常见任务时，前 3 条结果内必须命中目标模块。
- 后续功能 PR 在涉及对应交互面时，必须同步更新 Wiki 或明确标记“此能力仅保留在 docsite”。

## 2. User Experience & Functionality

### User Personas

- 新用户：第一次打开 MPE，需要快速理解界面、节点、连接、导入导出与基本操作。
- 日常资源开发者：长期在编辑器、工具箱、调试工作台中工作，需要在具体界面旁获取简明帮助。
- 迁移用户：已有 Pipeline 或 YAMaaPE 项目，需要确认兼容要求、导入方式、自动布局与常见坑。
- 集成/开发者：需要安装说明、嵌入协议、开发协作说明，这类内容适合保留在外部文档站长期阅读。

### User Stories

- As a 新用户, I want to 在编辑器内看到精简的上手路径 so that 我不需要在多页外部文档之间来回切换。
- As a 资源开发者, I want to 从当前界面直接打开对应 Wiki 模块 so that 我能边操作边确认正确做法。
- As a 调试用户, I want to 快速找到前置条件、运行方式和排障说明 so that 我能更快定位问题。
- As a 迁移用户, I want to 看到兼容清单和导入注意事项 so that 我能判断旧项目是否适合进入 MPE。
- As a 集成/开发者, I want to 保留完整外部文档入口 so that 长篇协议、安装命令和开发说明不会被塞进不适合阅读的 Modal。

### Acceptance Criteria

- Wiki 首页必须按“用户任务”组织，不直接复刻 docsite 的目录树。
- 单个 Wiki 模块建议控制在 3-7 个步骤；超长内容必须拆成多个模块。
- 每个迁入 Wiki 的模块都必须可搜索、可分享、可在至少一个真实界面入口中到达。
- 高价值界面入口至少覆盖：全局工具栏、调试工作台、工具箱、字段面板、连接面板、Pipeline 面板、导入导出区域、AI 相关区域。
- Wiki 中的术语必须与当前产品语义一致，例如“调试工作台”而不是“调试模式”。
- 首批上线允许使用文字为主的模块；图片/视频是增强项，但高频模块必须预留媒体位置或替代性步骤说明。
- 对于保留在 docsite 的长文，应用内至少要有摘要入口或外链，不允许“只删不导”。

### Non-Goals

- 不做 VitePress 页面的一比一机械搬运。
- 不在首期把安装命令、平台部署、嵌入协议、开发者协作说明全部迁入 Wiki Modal。
- 不在首期建设完整 CMS、WYSIWYG 编辑器或通用 Markdown 文档引擎。
- 不在本 PRD 中决定立即下线 docsite；首期目标是完成内容分流与迁移，而不是站点归零。
- 不要求本轮立即恢复当前被隐藏的 Wiki 模块入口；入口恢复应以内容覆盖率与实现进度为准。

## 3. AI System Requirements (If Applicable)

### Tool Requirements

本次 PRD 对产品运行时不引入新的 AI 依赖。Wiki 搜索与阅读继续使用当前非 AI 方案：

- 内容组织：`Entry -> Module -> Step -> Block`
- 搜索方式：标题、摘要、关键词、`searchText` 的本地匹配
- 展示方式：`WikiModal`、上下文触发器、哈希分享链接

若后续使用 AI 辅助撰写 Wiki 内容，只允许其作为“初稿生成工具”，不得跳过人工校对，不得把 AI 产出直接视为产品文档事实来源。

### Evaluation Strategy

对于 AI 参与的作者工作流，评估标准如下：

- 源文一致性：迁移内容必须可回溯到现有 docsite 或现行产品行为。
- 术语一致性：与当前产品 UI 和交互语义一致，不得引入旧名词或已移除能力。
- 任务可执行性：按照步骤执行时，用户应能在现有界面中找到对应入口。
- 漏项检查：高频页面迁移后，不得丢失前置条件、注意事项、排障入口。

## 4. Technical Specifications

### P0 Boundaries

P0 仅做文档治理落地，交付物限定在 `dev/design/`：

- 允许修改：`wiki-docsite-migration-prd.md` 与配套迁移清单文档。
- 禁止修改：`src/`、`LocalBridge/`、`docsite/docs/`、`dev/instructions/`。
- 禁止事项：不新增 Wiki 条目、不改搜索索引、不改分享链接格式、不调整入口组件、不迁移图片/视频资源。
- P0 的“落地”定义为：冻结迁移规则、页面去向矩阵、首批模块级清单、上下文入口清单、P1 最小边界，并写入执行记录。

### Architecture Overview

当前内置 Wiki 已具备可复用的最小体系：

- 内容元数据：`src/wiki/registry.ts`
- 内容模型：`src/wiki/types.ts`
- 搜索索引：`src/wiki/searchIndex.ts`
- 分享链接：`src/wiki/wikiUrl.ts`
- 状态管理：`src/stores/wikiStore.ts`
- 阅读器与触发器：`src/features/wiki/components/WikiModal.tsx`、`WikiPonderTrigger.tsx`

迁移后的推荐内容流为：

1. 以现有 docsite 页面做内容清点与分流。
2. 将 `Wiki First` / `Hybrid` 内容重写为任务导向模块，而不是逐页照搬。
3. 按现有 Wiki 类型定义产出 `WikiModule` 与 `WikiModuleSearchIndex`。
4. 在 `registry.ts` 中注册条目，在 `searchIndex.ts` 中纳入搜索。
5. 将高频模块挂到具体界面的 `WikiPonderTrigger` 或全局入口。

Wiki 内容粒度规则如下：

- `Entry`：一个主题域，例如“工作流编辑”“调试”。
- `Module`：一个完整使用场景，例如“连接面板”“从节点运行”“ROI 工具”。
- `Step`：一个原子任务或一个问题单元，例如“打开连接面板”“只执行动作”“理解负坐标”。
- `Block`：段落、提示、代码、图片、视频或内联组件。

建议的编写规则：

- 一个 `Step` 只解决一个任务或一个认知问题。
- 一个 `Module` 内优先保持同一上下文，不跨多个主界面乱跳。
- 长表格、长命令清单、完整协议字段表默认不放入 Wiki，而是保留 docsite 或拆成摘要 + 外链。

### Routing Rules

P0 将页面归属冻结为以下四类，后续实现不得自行改口径：

- `Portal`：仅承载首页、目录、总入口，不进入 Wiki Modal，不要求一比一迁入。
- `Wiki First`：高频操作型内容以 Wiki 为主责载体，docsite 后续可收敛为补充版或历史版。
- `Hybrid`：Wiki 负责应用内摘要、入口、前置条件与常见任务；docsite 保留长文、矩阵、命令、完整背景。
- `Docsite First`：保留在外部文档站长期阅读，应用内仅提供一句话摘要与外链，不进入首批迁移主线。

推荐判定法如下：

- 满足“强上下文 + 高频操作 + 需要在界面旁就地查看”的页面，归 `Wiki First`。
- 满足“需要应用内摘要，但完整内容过长或依赖大表格/命令/背景说明”的页面，归 `Hybrid`。
- 满足“开发者文档、安装部署、协议参考、机制解释、长篇背景”为主的页面，归 `Docsite First`。
- 满足“站点门户、导航、总目录”角色的页面，归 `Portal`。

### Wiki Template Freeze

P0 冻结 Wiki 内容模板规则如下：

- 命名规则：`Entry` 使用主题域命名；`Module` 使用单一场景命名；避免把多个主界面揉进一个模块。
- 摘要规则：每个 `Module` 必须提供一句面向任务的 `summary`，描述用户为什么打开它，而不是复述页面标题。
- 步骤粒度：单个 `Module` 建议 3-7 个 `Step`；P1 只冻结到模块级，不在 P0 提前写死最终 step 文案。
- 关键词规则：关键词优先使用当前产品 UI 叫法、常见中文搜索词和必要英文术语，避免旧名词。
- `searchText` 规则：只补充标题与关键词不足以覆盖的自然搜索表达，不重复堆砌同义词。
- 媒体策略：P1 文本优先；图片仅复用现有公开演示素材；视频只给高价值模块；P0 不迁任何图片/视频文件。
- 术语规则：中文主词优先，必要时补英文括注；保持与当前产品语义一致，例如“调试工作台”“本地能力”“会话”。
- 外链规则：`Hybrid` 与 `Docsite First` 模块必须保留外部长文落点；不允许把关键信息写成“请自行查看 docsite”而不给目标。

### Integration Points

迁移实施时需要重点关注以下集成点：

- 内容注册：`src/wiki/registry.ts`
- 搜索维护：`src/wiki/searchIndex.ts`
- 分享链接：`src/wiki/wikiUrl.ts`
- Modal 首页与目录：`src/features/wiki/components/WikiModal.tsx`
- 上下文入口：`src/features/wiki/components/WikiPonderTrigger.tsx`
- 已存在入口：`GlobalPanel`、`DebugModal`、`ToolboxPanel`
- 待扩展入口：字段面板、连接面板、Pipeline 面板、导入导出入口、AI 相关面板

P0 对上下文入口分层冻结如下：

- 已存在入口：`GlobalPanel`、`DebugModal`、`ToolboxPanel`。
- P1 首批必须接入：字段面板、连接面板、Pipeline 面板、导入导出区域。
- P2 扩展入口：AI 相关区域、空画布/首次使用提示、旧文件首次打开/迁移提示、更多 LocalBridge 失败场景。

需要预先处理的工程问题：

1. 当前 `registry.ts` 与 `searchIndex.ts` 都是手工维护，模块数量上升后容易漏注册、漏索引。
2. 当前 Wiki 媒体使用 `src` 字符串直连，docsite 中依赖 `/docs/` Base 的资源路径不能直接照搬。
3. 当前 Wiki 条目较少时适合手写；若迁移模块达到 20+，应考虑引入统一 manifest 或生成脚本，避免双份维护。

内容资产策略建议：

- 文本优先：首批模块先保证结构、搜索、入口与术语正确。
- 图片复用：优先复用现有演示截图，但必须改成主应用可访问的资源路径。
- 视频慎用：仅为高价值模块提供短视频，不追求所有页面都带动图。
- 代码块摘要化：命令行安装、协议 JSON 示例保留最短必要片段，长示例仍链接 docsite。

### Security & Privacy

Wiki 迁移需要遵守以下边界：

- 哈希分享链接只允许包含 `entryId/moduleId/stepId`，不得带本地路径、设备标识、截图内容、会话数据。
- Wiki 内容不得嵌入用户当前运行时的真实配置值，例如本地目录、OCR 路径、API Key。
- 若使用静态截图或视频，素材必须是可公开的演示资源，不能含敏感账号、密钥、路径或设备信息。
- 当 Wiki 涉及 AI API 配置说明时，仍需保留对本地明文存储/CORS 等风险的明确提示。

## 5. Risks & Roadmap

### Phased Rollout

#### P0: 内容分流与模板冻结

- 冻结 docsite 页面去向矩阵。

#### P1: 最小可用 Wiki（高频操作闭环）

- 扩展现有 `debug` 与 `toolbox` 条目，优先完成调试与字段快捷工具。
- 新增 `workflow` 与 `start` 条目，覆盖节点、字段、连接、导入导出、快速上手。
- 保证全局首页、搜索、分享可用。
- 此阶段 docsite 继续保留为完整外部文档。

P1 最小边界冻结如下：

- 只做高频操作闭环，不碰安装、协议、开发协作、长篇机制文档。
- 模块冻结到 `Entry / Module` 层，不要求在 P1 开始前写死每个 `Step` 文案。
- 首批必须优先覆盖：调试、字段快捷工具、节点、字段、连接、导入导出、LocalBridge 基础连接说明、基础迁移说明。
- `Hybrid` 页面只做摘要 + 跳转，不追求与 docsite 双端等量搬运。

阶段门槛：

- 用户可在应用内完成“编辑基础操作 + 工具箱 + 调试”的主要查阅闭环。
- Wiki 至少覆盖 12 个以上高频模块。

#### P2: 上下文化与迁移场景补全

- 新增 `localbridge`、`ai`、`migrate` 条目。
- 将字段面板、连接面板、导入入口、AI 相关界面接入上下文触发器。
- 为 `Hybrid` 页面补齐摘要模块与外部长文跳转关系。

阶段门槛：

- 新用户、迁移用户、调试用户三类核心人群都能在应用内找到对应的第一入口。
- docsite 中高频操作型页面可逐步转为“详细版/补充版”。

#### P3: 文档治理与角色收口

- 清理 docsite 中与 Wiki 重复且不再需要维护的段落，保留外部站点应有内容。
- 明确长期职责边界：Wiki 负责“应用内操作与排障”，docsite 负责“安装、部署、协议、开发、长篇参考”。
- 评估是否恢复当前被隐藏的 Wiki 入口为默认可见，并补齐后续维护流程。

#### P4: 可维护性增强（可选）

- 评估是否将 Wiki 注册、搜索索引与元数据从手工维护升级为 manifest 或生成式流程。
- 评估是否为内容作者引入更轻量的 Markdown/JSON 源格式，而不是持续手写 TS。

### Technical Risks

- 内容漂移：同一能力若同时存在于 Wiki 与 docsite，后续容易只改一边。
  - 缓解：为每个页面明确主责载体，`Hybrid` 页面只保留摘要，不保留双份完整说明。
- 注册遗漏：当前 Wiki 需要手工维护注册与搜索索引，模块增多后容易漏项。
  - 缓解：P1 前至少制定内容清单校验规则，P4 再考虑 manifest / codegen。
- 资源路径失效：docsite 资源引用与主应用资源基址不同，直接复用会导致媒体加载失败。
  - 缓解：迁移时统一整理媒体路径，不允许直接复制依赖 `/docs/` 的地址。
- Modal 阅读负担：协议、安装命令、大段矩阵表格不适合放在内置 Wiki。
  - 缓解：坚持 `Docsite First` 边界，只在 Wiki 中提供摘要和跳转。
- Wiki 当前被隐藏：若内容覆盖率不足就恢复入口，会造成“有壳无料”的负体验。
  - 缓解：以 P1 模块覆盖率为恢复开关，而不是先恢复入口再补内容。

### Open Questions

- Wiki 长期是否要成为“操作型文档唯一真源”，还是继续与 docsite 双源并存。
- Wiki 内容源是否继续使用 TS 手写，还是引入更适合文档维护的中间格式。
- 外部文档站未来是否保留完整“指南”栏目，还是收敛为“安装/协议/开发者中心”。
