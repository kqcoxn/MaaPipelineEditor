# MaaPipelineEditor 门面站点建设规划

## 一、方向结论

当前这版规划的优点是完整、稳妥、易落地，但问题也很明显：

- 更像一套“常规产品官网”模块清单，而不是有记忆点的产品叙事
- `Features / Showcase / Quick Start` 彼此平级，主视觉焦点不够集中
- “AI 时代产品感”主要停留在文案层，缺少界面表达方式上的升级
- “亮点特性”还是卡片式罗列，没有把“实践流程”演出来

本次调整后的核心方向：

- 页面主结构不再是“很多普通区块”，而是“Hero + 多组 Feature Story Slides + 信任与转化收口”
- 主功能采用**多组 slide** 讲述亮点特性的完整实践流程，每组 slide 对应一个真实任务
- 视觉语言从“传统科技官网”升级为“工作流 + 智能辅助 + 实时反馈”的 AI 产品风格
- 展示方式要更丰富，加入滚动叙事、分镜切换、前后对比、节点聚焦、状态浮层、结果回显等表现手法

核心信息更新为：

> 不是在展示一个编辑器，而是在展示“如何更快地理解、构建、调试、分享 Pipeline”。

## 二、产品定位

### 2.1 目标

为 MaaPipelineEditor (MPE) 打造一个更现代、更有产品感的门面站点，用于：

- 让第一次访问的用户 10 秒内看懂产品价值
- 让潜在用户通过几组流程 slide 迅速建立“我也能这样做”的感觉
- 突出可视化编辑、流程理解、协作分享、AI 辅助等关键能力
- 提升转化率：在线使用、查看文档、GitHub Star、加入讨论

### 2.2 目标用户

- MaaFramework 资源开发者
- 自动化流程开发者
- 需要维护或理解复杂 Pipeline 的协作者
- 想用更低门槛方式阅读、修改、调试流程的用户

### 2.3 新风格定位

- 关键词：智能工作流、实时反馈、未来工具感、可操作、可信赖
- 避免：普通 SaaS 官网、单纯卡片堆砌、过度炫技但不说明能力
- 强调：流程可视化、信息逐步展开、局部高亮、机器辅助感、操作结果即时出现

### 2.4 主传播语

- 主标题候选：让 Pipeline 从代码负担，变成可视化工作流
- 副标题候选：导入、理解、编辑、调试、分享，一套界面完成
- 核心短句：告别手调千行 JSON，用工作流方式构建 MaaFramework Pipeline

## 三、信息架构重构

### 3.1 页面结构

新的页面结构建议如下：

1. 固定导航栏
2. Hero 首屏
3. 信任条 / 实时状态条
4. Feature Story Slides 主模块
5. 能力总览补充区
6. 生态与技术可信度
7. 快速开始
8. 最终 CTA
9. 页脚

### 3.2 结构原则

- **主角前置**：首屏之后立刻进入多组 slide，而不是先铺一排普通 feature 卡片
- **流程先于功能**：先让用户看到“怎么做成一件事”，再总结“背后有哪些能力”
- **大模块少而强**：减少重复区块，把展示资源集中到 1 个核心舞台
- **滚动有剧情**：页面不是往下堆内容，而是像产品发布页一样逐段推进认知

## 四、核心模块：多组 Feature Story Slides

### 4.1 模块定义

这是整个站点的主功能区，也是最重要的展示模块。

每一组 slide 代表一个“亮点特性的实践流程”，不是单张海报，也不是单个功能卡片，而是：

- 一个真实任务场景
- 一个清晰目标
- 3 到 5 张连续 slide
- 每张 slide 展示一个关键操作或一个关键结果
- 用户滚动或点击时，舞台内容发生变化，形成完整叙事

### 4.2 每组 slide 的标准结构

每组建议包含以下信息：

- `场景标题`：这组流程解决什么问题
- `一句话结果`：最终帮用户节省了什么成本
- `步骤 slide 1`：问题进入点 / 原始状态
- `步骤 slide 2`：关键操作
- `步骤 slide 3`：智能反馈 / 自动化辅助 / 可视化变化
- `步骤 slide 4`：结果输出 / 协作 / 调试成功
- `补充信息`：关联能力标签、对应截图、短说明

### 4.3 交互形式建议

优先使用以下其中一种：

#### 方案 A：纵向滚动驱动的横向分镜

- 页面纵向滚动
- 当前组 slide 区域 sticky 固定
- 右侧主舞台横向切换不同 slide
- 左侧同步更新步骤文案与标签

适合做出更强的“发布会式”展示效果，是首选方案。

#### 方案 B：分组 Tab + 组内 slide 切换

- 顶部先切换不同亮点场景
- 进入组内后，再左右切换 slide
- 对实现复杂度更友好

适合第一版较快落地，但氛围感稍弱。

### 4.4 每张 slide 的内容构成

建议统一为以下三栏逻辑：

- 左侧：步骤编号、标题、两行说明、能力标签
- 中间：主视觉舞台，展示界面局部、节点流、状态高亮
- 右侧：结果说明、前后对比、提示浮层、指标变化

在移动端改为上下堆叠：

- 先显示标题与步骤
- 再显示舞台图
- 最后显示结果卡片

### 4.5 推荐的亮点流程分组

建议至少做 4 组，首发就能形成“展示花样多”的观感。

#### Group 1：从 JSON 到可视化工作流

目标：让第一次接触产品的用户立刻看懂最核心价值。

slide 建议：

1. 粘贴或导入现有 Pipeline JSON
2. 自动解析成节点图与路径关系
3. 聚焦关键节点并编辑字段
4. 导出结果，完成从文本到流程图的切换

重点表现：

- 文本到图形的转化动画
- 节点自动布局
- 被高亮的关键路径
- Before / After 对比

#### Group 2：快速看懂陌生项目逻辑

目标：突出“理解成本降低”这个痛点价值。

slide 建议：

1. 打开一个复杂、陌生的 Pipeline
2. 自动布局与节点搜索帮助快速定位
3. 路径高亮、节点聚焦、字段联动查看
4. 几分钟内理清主流程与依赖关系

重点表现：

- 杂乱流程图逐步变清晰
- 搜索命中节点发光
- 关键链路被单独提取
- 边看边理解的“辅助感”

#### Group 3：从编辑到调试闭环

目标：展示不是“只能画图”，而是能完成工作闭环。

slide 建议：

1. 修改节点参数
2. 启动本地服务或调试能力
3. 截图、日志、状态回显同步出现
4. 定位问题并修正，完成验证

重点表现：

- 配置面板和运行反馈联动
- 调试状态条 / 小窗回显
- 失败与修复的前后结果

#### Group 4：模板复用与团队协作

目标：突出生产效率和协作价值。

slide 建议：

1. 从模板创建新流程
2. 调整局部节点适配当前项目
3. 导出、分享、持久化配置
4. 团队成员快速复用并继续编辑

重点表现：

- 模板卡进入编辑舞台
- 配置差异被可视化标记
- 分享结果卡、导出格式、兼容提示

#### Group 5：AI 辅助能力预告区

目标：建立“面向未来”的产品想象力，但不夸大当前状态。

slide 建议：

1. 用自然语言描述想做的流程
2. AI 推荐节点或补全字段
3. 自动给出可疑配置提示
4. 用户确认后落到可编辑工作流

重点表现：

- Prompt 输入条
- AI 建议卡片流入工作区
- 风险提示与确认面板

说明：

- 如果这部分暂时未完全落地，可明确标记为 `Coming Soon`
- 不建议把未实现能力写成已完成，应采用“未来路线”表达

### 4.6 每组 slide 的表现形式要有变化

为了避免“只是同一种大卡片重复四次”，每组应该有不同视觉节奏：

- 一组用 Before / After 强对比
- 一组用路径聚焦与聚光灯效果
- 一组用操作面板 + 结果回显
- 一组用模板流转与协作结果卡
- 一组用 AI 输入条 + 建议浮层

也就是说，统一框架，变化舞台语言。

### 4.7 这块替代哪些旧模块

这组 Feature Story Slides 会替代原规划中大部分：

- `Features`
- `Showcase`
- 一部分 `Quick Start`

这些内容不再分散出现，而是合并为一个更强的主叙事模块。

## 五、页面详细结构

### 5.1 导航栏

- 左侧：Logo + MaaPipelineEditor
- 中间：亮点流程、能力、文档、GitHub
- 右侧：在线体验、查看文档
- 风格：半透明、毛玻璃、滚动后边框与阴影增强
- 移动端：抽屉菜单，CTA 保留在首层

### 5.2 Hero 首屏

#### 目标

在第一屏就建立“这是一个 AI 时代的流程工具”印象。

#### 内容建议

- 左侧：
  - 版本 Badge
  - 强主标题
  - 一句副标题
  - 两个 CTA：在线使用 / 查看文档
  - 一条简短信任信息：开源、跨平台、可视化编辑
- 右侧：
  - 一个“活的舞台”
  - 不是静态截图，而是由节点、路径、浮层、AI chip、状态条构成的演示场景

#### 首屏舞台推荐表达

- 中间是一个半真实的编辑器画布
- 左上浮现 Prompt/导入条
- 中间节点逐步排开
- 右侧参数面板出现字段变更
- 底部出现状态流或调试结果条

也就是让用户在首屏就看到“输入 -> 工作流 -> 反馈”。

### 5.3 信任条 / 实时状态条

这里不建议只做一排死板数字，建议改成更产品化的状态带。

可包含：

- GitHub Stars
- 最近版本号
- 支持在线使用 / 本地增强
- 支持模板 / 调试 / 可视化

表现形式：

- 横向滚动标签条
- 或 4 个微卡片，带图标和短句

### 5.4 Feature Story Slides

这是页面核心，详见第四章。

### 5.5 能力总览补充区

在看完流程后，再用一组能力卡片做总结。

建议从原本 6 个 feature 卡片压缩为 4 类能力：

1. 可视化编辑
2. 理解与分析
3. 调试与验证
4. 模板与协作

这里的作用不是重新讲故事，而是帮助用户做快速扫描。

### 5.6 生态与技术可信度

不必把技术栈做成大段独立区块，可压缩为以下形式：

- React / TypeScript / React Flow / Go 等技术徽章
- 开源、可部署、与主项目一致的工程栈说明
- 一段简短描述：为什么这个站点和主项目体验可信

### 5.7 快速开始

保留，但变得更短、更像转化引导。

建议只保留三步：

1. 打开在线编辑器
2. 导入或新建 Pipeline
3. 开始编辑并调试

搭配一个极短代码片段或命令片段即可，不要喧宾夺主。

### 5.8 最终 CTA

建议分成两个动作层级：

- 主 CTA：立即在线体验
- 次 CTA：阅读文档 / 去 GitHub Star

文案建议更有行动感：

- 现在就把 Pipeline 拉进可视化工作流里
- 用更少的理解成本，完成更多流程编辑

## 六、视觉语言升级

### 6.1 总体气质

目标不是“更炫”，而是“更像现代智能工具”。

关键词：

- 深色基底
- 低饱和工业感
- 青蓝高亮
- 局部荧光绿作为状态色
- 大面积留白与局部信息密度对比
- 毛玻璃、渐变雾面、微网格、路径发光

--bg-elevated: #122033;
--text-primary: #f5f7fb;
--text-secondary: #9fb0c3;

--border: rgba(164, 184, 204, 0.18);

#### 浅色主题

### 6.3 字体建议

避免纯系统默认风格，建议更有产品识别度：

- 标题字体：`Sora` 或 `Space Grotesk`，中文回退 `Noto Sans SC`
- 正文字体：`IBM Plex Sans` 或 `Noto Sans SC`
- 代码字体：`JetBrains Mono`

如果要更稳妥，也可以：

- 标题：`HarmonyOS Sans SC`
- 正文：`Noto Sans SC`
- 代码：`JetBrains Mono`

### 6.4 背景层次

建议同时使用 3 层背景：

1. 深色渐变底
2. 极淡网格或坐标纸纹理
3. 局部发光云团 / 柔和光斑

这样页面会更像“工作台”，而不是普通网页背景色。

### 6.5 卡片与容器风格

- 主卡片：半透明暗底 + 1px 弱边框 + 轻微模糊
- 舞台区：更深一层，像工作台屏幕
- 浮层：适当抬高亮度，像实时状态提示
- 圆角：`20px - 28px`
- 阴影：柔和、扩散，不做过黑重阴影

## 七、动效设计

### 7.1 动效原则

- 动效服务于理解，不服务于炫耀
- 动效要突出“流程变化”“状态切换”“智能反馈”
- 重点做舞台内容变化，少做无意义飘动

### 7.2 推荐动效清单

#### Hero

- 节点轻微入场
- 连线逐步绘制
- 参数面板延迟出现
- 状态条从底部滑入

#### Feature Story Slides

- slide 切换时主舞台做局部过渡，不是整屏硬切
- 当前步骤对应元素高亮，上一状态柔化
- Before / After 对比时支持遮罩滑动
- 搜索命中时节点脉冲发光
- 调试结果出现时状态点变色

#### 细节反馈

- 标签 hover 出现微光边框
- 按钮 hover 做轻微抬升和渐变偏移
- 数字与版本信息可做轻量滚动计数

### 7.3 推荐实现方式

- 基础入场与局部切换：`Framer Motion`
- 分镜舞台：`CSS scroll-snap` + `sticky` + `useScroll`
- 线条绘制与节点高亮：SVG / Canvas 按需使用

## 八、内容与组件设计

### 8.1 建议新增的数据驱动结构

为了让“多组 slide”可维护，建议用内容配置驱动页面，而不是把每一组硬编码在 JSX 里。

建议新增：

```ts
type FeatureStoryGroup = {
  id: string
  eyebrow: string
  title: string
  summary: string
  outcome: string
  tags: string[]
  slides: FeatureSlide[]
}

type FeatureSlide = {
  id: string
  step: string
  title: string
  description: string
  layout: "stage" | "compare" | "focus" | "panel" | "result"
  media: {
    type: "image" | "video" | "mock"
    src?: string
  }
  callout?: string
  metrics?: string[]
}
```

这样后续只要改内容数据，就能继续扩充新的流程组。

## 九、技术方案

### 9.1 技术栈

| 层级     | 技术选型       | 说明                     |
| -------- | -------------- | ------------------------ |
| 构建工具 | Vite           | 与主项目保持一致         |
| 前端框架 | React 19       | 与主项目保持一致         |
| 类型系统 | TypeScript 5.8 | 保持类型安全             |
| 样式方案 | Tailwind CSS 4 | 配合 CSS 变量构建主题体系 |
| 动画库   | Framer Motion  | 负责滚动和分镜过渡        |
| 图形层   | SVG / Canvas   | 用于连线、光效、舞台装饰  |
| 图标     | Lucide React   | 简洁现代                 |
| 部署     | GitHub Pages   | 首发部署简单稳定         |

### 9.2 技术重点

这次真正的技术重点不在“普通官网组件”，而在：

- slide 分组数据结构
- scroll-driven story 组件
- 高质量图片与动效的平衡
- 响应式下的舞台降级方案

### 9.3 响应式策略

- 桌面端：完整分镜舞台 + sticky 叙事
- 平板端：组内 slide 简化，保留重点动效
- 移动端：改为竖向卡片顺序播放，弱化复杂横向位移

原则：

- 不强行在移动端复刻桌面舞台
- 要保证信息清晰和性能优先

## 十一、资源清单

### 11.1 需要准备的内容

- 产品主标题与副标题
- CTA 链接
- 最新版本号
- GitHub Stars 数据来源
- 文档入口
- 在线体验入口

### 11.2 需要准备的视觉素材

- 首页 Hero 主舞台截图或录屏
- 4 到 5 组流程 slide 对应截图
- 模板、调试、导出、搜索等局部画面
- Logo 与 favicon
- Open Graph 分享图

### 11.3 风险点

- 如果每组 slide 都依赖大 GIF，性能会明显变差
- 如果没有统一的视觉框架，容易做成“很多风格不同的大卡片”
- 如果 AI 能力尚未落地，文案必须明确区分“现有能力”和“规划中能力”

---

## 十二、落地原则

1. 首屏之后立即进入主叙事，不要让用户滚很久才看到核心价值
2. 多组 slide 是站点主角，普通 feature 卡片只能做补充
3. 一组 slide 讲一个任务闭环，不讲空泛功能点
4. 视觉要有未来工具感，但必须服务理解
5. 移动端允许降级，但不能丢失主线
6. AI 表达要克制，强调“辅助提效”，不要写成空泛口号
