# MPE 流程探索人机协作模式

## 1. 背景与动机

### 1.1 当前痛点

在 MaaFramework Pipeline 的开发过程中，开发者通常会遇到以下问题：

- **古法编写效率低**：需要手动截图、逐个配置节点、反复调试验证
- **AI 辅助视野有限**：现有 AI 补全大多停留在单节点或字段级别，缺少“流程下一步该做什么”的整体判断
- **验证周期长**：往往要把一串步骤写完后再整体运行，问题发现得晚、回溯成本高
- **维护成本高**：一旦游戏 UI 改版，旧流程容易局部失效，但排查通常只能靠人工逐步试错

### 1.2 核心洞察

借鉴 RLHF（Reinforcement Learning from Human Feedback）中的“人在回路”思想，但不追求自动训练本身，而是强调协作闭环：

> AI 负责探索与提议，人负责确认与纠偏，系统负责承载可视化审阅、执行与回退。

这里的关键不是让 AI 一步到位“自动写完流程”，而是让它在每一步都提供足够有用、足够可审阅的建议，帮助用户更快走到正确结果。

### 1.3 设计原则

- **先提议，后落地**：AI 产出的是待确认提议，不直接写入正式流程
- **每步可审阅，可回退**：每次探索都应可观察、可修改、可撤销
- **结构化优先**：AI 输出应尽可能是稳定的数据结构，而非自由文本
- **人始终拥有最终决定权**：涉及歧义、高风险、不可逆动作时必须保留人工把关
- **与现有工作流共存**：探索模式增强编辑体验，不替代现有的字段编辑、调试、模板与手工编排能力

## 2. 模式定义

### 2.1 什么是流程探索人机协作模式

这是一种**流程级的人机协作工作流**，其中：

| 角色 | 职责 |
| --- | --- |
| **AI** | 根据当前画面、历史上下文和用户目标，提议“下一步该做什么”，包括识别方式、动作类型、参数和备选方案 |
| **人** | 在 MPE 中审阅 AI 的提议，进行确认、调整、否决、回退或手动接管 |
| **MPE** | 提供可视化交互环境，承载探索会话、截图标注、Ghost Node 预览和流程落地 |
| **LocalBridge / mfw** | 提供截图、识别、点击、滑动、输入等底层能力，负责将确认后的步骤真正执行到设备上 |

### 2.2 核心交互闭环

```text
用户设定目标
      ↓
LocalBridge 获取当前画面
      ↓
AI 结合目标 + 历史上下文 → 决策下一步动作
      ↓
MPE 可视化呈现（Ghost Node + 截图标注 + 决策摘要）
      ↓
人审阅并调整 → [确认] [修改] [拒绝] [回退] [终止]
      ↓
确认后 → 节点正式落地 → 执行该步骤
      ↓
执行结果 + 新画面回流 → 下一轮探索
```

### 2.3 范围边界

本模式：

- 不是“全自动无人值守脚本生成器”
- 不是对普通编辑模式的替代，而是其增强层
- 不是单节点 AI 补全的简单放大版，而是带有会话、状态和执行闭环的流程协作模式

## 3. 目标用户与典型场景

### 3.1 场景一：新手快速上手

**用户**：刚接触 MaaFramework 的新开发者  
**目标**：完成游戏登录流程

**流程**：

1. 用户在 MPE 中输入目标：“完成游戏登录”
2. AI 分析当前游戏画面，提议第一步：“点击账号输入框”
3. MPE 在画布上显示半透明 Ghost Node，并在截图中高亮推荐区域
4. 用户确认位置正确，或微调 ROI 后点击“确认”
5. 节点变为正式节点，LocalBridge 执行该步骤
6. 画面切换到输入界面，探索继续推进

**价值**：新手无需先理解 MaaFramework 的字段细节，也能在引导下完成首个可运行 Pipeline。

### 3.2 场景二：复杂流程拆解

**用户**：有经验的开发者  
**目标**：完成游戏内一个多步骤活动

**流程**：

1. 用户输入目标：“完成每日签到活动”
2. AI 识别出关键路径：点击活动入口 → 切换签到标签 → 点击签到按钮 → 关闭奖励弹窗
3. 每一步都在 MPE 中以可审阅提议呈现，用户可根据实际情况微调
4. 遇到分支时，例如“今日已签到”或“活动入口位置变化”，AI 给出多个候选路径并提示用户选择

**价值**：把复杂流程中的“探索脑力”转移给 AI，人主要负责异常情况判断与结果把关。

### 3.3 场景三：流程维护与适配

**用户**：需要维护已有 Pipeline 的开发者  
**目标**：适配游戏 UI 改版

**流程**：

1. 用户加载旧版 Pipeline，启动探索模式
2. AI 对比旧节点配置与当前画面，识别失效步骤或高风险步骤
3. 对失效步骤重新提议识别区域与动作参数
4. 用户逐条审阅并确认新的适配方案

**价值**：降低 UI 改版后的维护成本，帮助用户更快定位失效点并完成修复。

### 3.4 场景四：人机接力编辑

**用户**：熟悉编辑器但不想被 AI 绑住流程的开发者  
**目标**：在关键步骤由 AI 提议，在细节步骤手工接管

**流程**：

1. 用户让 AI 先完成主路径探索
2. 在某个复杂节点处切换到普通编辑模式，手动调整参数
3. 保存后继续探索，AI 将手动编辑结果作为新的上下文

**价值**：探索模式不会“锁死”用户，而是可以自然退回现有编辑工作流。

## 4. 产品目标与阶段范围

### 4.1 产品目标

- 降低从“有目标”到“拿到首个可运行流程”的门槛
- 缩短流程编写与验证之间的反馈回路
- 降低复杂流程和维护场景中的认知负担
- 把 AI 能力从字段补全提升到流程级辅助

### 4.2 MVP 范围

MVP 先聚焦最小闭环，只解决“能稳定跑起来”的核心问题：

- 单控制器、单活跃探索会话
- 单步提议 → 人确认或修改 → 执行 → 下一步
- 支持基础动作类型：点击、滑动、输入、等待、判断
- 支持截图标注、Ghost Node 预览、决策摘要展示
- 支持回退一步、拒绝当前提议、终止探索
- 支持在普通字段面板中对当前提议进行编辑后再确认

### 4.3 暂不纳入 MVP 的内容

- 多控制器并行探索
- 长链路全自动无人值守生成
- 大规模并行分支树生成
- 对旧 Pipeline 的全量自动迁移
- 基于用户历史自动学习个人偏好模型

### 4.4 阶段演进建议

| 阶段 | 目标 | 核心能力 |
| --- | --- | --- |
| **Phase 1** | 打通探索闭环 | 会话管理、单步提议、Ghost Node、确认执行、回退 |
| **Phase 2** | 提升复杂场景可用性 | 分支候选、失败恢复、维护适配、模板辅助 |
| **Phase 3** | 形成知识沉淀能力 | 会话回放、模板沉淀、偏好记忆、团队共享 |

## 5. 交互与状态设计

### 5.1 探索会话模型

探索模式应被建模为一个独立的**Explore Session**，至少包含以下状态项：

| 状态项 | 说明 |
| --- | --- |
| **session_id** | 当前探索会话唯一标识 |
| **controller_id** | 当前关联的设备或控制器 |
| **goal** | 当前探索目标描述 |
| **status** | 会话状态，例如 idle / waiting_llm / reviewing / executing |
| **current_step** | 当前正在处理的第 N 步 |
| **proposal** | 当前待审阅的 AI 提议 |
| **screenshot_context** | 当前步骤关联的截图、ROI、标注信息 |
| **confirmed_chain** | 已确认并落地的步骤序列 |
| **history** | 用户确认、修改、拒绝、回退等操作记录 |
| **last_error** | 最近一次失败信息或警告 |

额外约束：

- 同一时刻一个控制器只允许存在一个活跃探索会话
- 会话状态应独立于普通画布编辑状态，避免污染既有文件编辑逻辑

### 5.2 探索状态机

需要显式定义探索模式的状态迁移，避免“UI 看起来能点，但后台语义不清”的情况：

| 状态 | 含义 | 允许迁移 |
| --- | --- | --- |
| **idle** | 尚未开始探索 | waiting_screenshot / terminated |
| **waiting_screenshot** | 正在获取当前画面 | waiting_llm / failed |
| **waiting_llm** | 正在等待 AI 决策结果 | reviewing_proposal / failed |
| **reviewing_proposal** | 用户正在审阅当前提议 | testing / executing / waiting_llm / paused / terminated |
| **testing** | 用户选择先测试当前步骤 | reviewing_proposal / executing / failed |
| **executing** | 已确认并正在执行步骤 | waiting_screenshot / failed / completed |
| **paused** | 用户手动接管或暂时中止 | reviewing_proposal / waiting_llm / terminated |
| **failed** | 当前步骤或会话失败 | reviewing_proposal / waiting_llm / terminated |
| **completed** | 目标达成，探索结束 | idle |
| **terminated** | 用户主动终止 | idle |

补充说明：

- 页面刷新或 WebSocket 重连后，若检测到未结束会话，可提供“恢复最近会话”入口
- 一旦控制器断连，会话应自动进入 `failed` 或 `paused`，而不是静默丢失

### 5.3 Ghost Node（幽灵节点）

Ghost Node 是 AI 当前提议的可视化承载物，推荐采用**“现有 Pipeline Node + Draft 元信息”**的方式落地，而不是直接新增一种可持久化节点类型。

原因：

- 现有画布节点体系已经稳定，正式节点的数据结构也已明确
- Ghost Node 本质上是“待确认渲染态”，不是最终存档态
- 未确认提议不应进入正式 Pipeline JSON，也不应污染正常的导出与保存逻辑

推荐表现：

- 半透明或降低饱和度
- 虚线边框，显式区分“待确认”
- 显示置信度、动作类型、关键识别信息
- 支持拖拽、框选调整、双击编辑字段
- 提供显式操作：确认、修改、拒绝、测试

### 5.4 截图联动标注

AI 的提议必须与截图强绑定，至少包含：

- 当前 `screenshot_id`
- AI 推荐的 ROI / target / begin / end 区域
- 候选目标框、置信度和标签
- 用户人工修正后的最终区域

交互要求：

- 用户可拖动调整框选范围
- 用户可重新框选并把结果回灌给 AI 或直接覆盖当前提议
- 一旦用户手动修改 ROI，该修改应视为当前步骤的权威输入

### 5.5 探索控制面板

建议在界面一侧提供固定控制面板，包含：

- **目标显示**：当前任务目标和控制器信息
- **进度显示**：第 N 步 / 预计总步数
- **决策依据摘要**：展示简短的 AI 选择理由与置信度，不展示冗长内部推理
- **操作区**：确认、修改、拒绝、回退、测试、终止
- **历史缩略图**：最近若干步截图与结果回顾
- **状态提示**：当前会话状态、错误信息、风险提醒

### 5.6 回退、接管与恢复

探索模式不仅要支持“向前走”，也必须明确“如何安全撤回”：

- **确认**：把 Ghost Node 正式写入当前流程，并记录到会话历史
- **回退一步**：撤销最近一次确认，同时通知后端回滚会话上下文
- **拒绝当前提议**：保留当前步骤号，但要求 AI 基于同一画面重新给出新方案
- **手动接管**：临时切回普通编辑逻辑，用户修改后再继续探索
- **恢复会话**：断连或刷新后，允许恢复最近会话的目标、步骤和历史截图

### 5.7 分支处理

MVP 阶段建议控制复杂度：

- 默认只展示 **1 个主提议 + 最多 2 个备选提议**
- 用户先在备选中做选择，再继续后续探索
- “一次生成完整分支树”放到后续阶段，而不是 MVP 强行覆盖

## 6. AI 决策与执行模型

### 6.1 决策场景分类

| 场景类型 | AI 行为 | MPE 呈现 |
| --- | --- | --- |
| **明确目标** | 识别到明确可交互元素，输出具体动作与参数 | 截图框选目标，生成 Ghost Node |
| **等待 / 判断** | 发现需要等待加载或判断状态 | 生成 Wait / Condition 类型提议，提示判断依据 |
| **分支选择** | 检测到多个可能路径 | 展示主提议和备选提议，等待用户选择 |
| **目标达成** | 检测到任务完成标志 | 提示结束探索并建议保存 |
| **异常 / 不确定** | 无法识别画面或置信度过低 | 提示人工介入，提供手动编辑入口 |
| **敏感 / 高风险** | 涉及不可逆或高风险操作 | 强制额外确认，必要时仅允许手动接管 |

### 6.2 AI 输出约束

为保证稳定性，AI 端输出应尽量是**结构化 JSON**，而不是自由文本。至少应包含：

- `proposal_id`
- `step_index`
- `action_type`
- `recognition_type`
- `node_draft`
- `overlays`
- `confidence`
- `reasoning_summary`
- `alternatives`
- `needs_human_attention`

其中：

- `node_draft` 应尽量贴合现有 Pipeline 节点结构
- `reasoning_summary` 只保留对用户有帮助的简短摘要，不暴露冗长内部推理
- `alternatives` 仅保留有限个候选方案，避免 UI 过载

### 6.3 置信度与人工介入规则

建议按置信度分层：

- **高置信度**：可以高亮推荐“确认”，但仍保留人工最终确认
- **中置信度**：默认引导用户检查 ROI 和关键参数
- **低置信度**：默认推荐人工重框选、修改或切换手动编辑

以下场景应直接要求人工介入，而不是继续自动推进：

- 屏幕元素识别冲突严重
- 动作目标不唯一
- 执行后画面变化与预期不一致
- 当前步骤涉及账号、密码、支付、删除、退出登录等敏感动作

### 6.4 安全边界

为了避免“AI 看起来聪明，但实际越界”：

- 涉及**不可逆动作**时必须显式二次确认
- 涉及**敏感输入**时应优先手动接管，或在用户明确授权后才执行
- 当 AI 无法解释为何选择某一步时，不应允许其默默执行
- 当设备状态异常、控制器断连、截图失败时，会话必须中断并提示用户

### 6.5 测试与执行策略

建议把“测试”与“正式执行”区分开：

- **先测试**：只验证当前提议在当前画面是否成立，适合高风险或低置信度步骤
- **正式执行**：在用户确认后，真正对设备发送动作
- **结果回流**：执行后要带回操作结果、执行前后截图、可能的异常信息

用户手动修改过的参数，应在当前步骤中优先于 AI 原始提议，并作为后续探索的新上下文。

## 7. 技术架构设计

### 7.1 总体原则

**决策**：探索模式由 LocalBridge 后端直接调用 mfw 与 LLM，不引入额外的 MCP 层。

原因：

- 探索会话本质上是“截图 → AI 提议 → 人确认 → 执行 → 新截图”的短闭环
- LocalBridge 已具备成熟的 mfw 调用链路，截图、点击、识别等能力可直接复用
- 后端更适合维护多轮上下文、缓存截图和管理 API Key
- 前端无需承担截图中转、模型调用和安全存储职责

这里更准确的表述应是：

> 执行链路是确定且可审计的，AI 决策是受约束的非确定性提议。

### 7.2 前后端职责划分

| 层级 | 模块 | 职责 |
| --- | --- | --- |
| **前端 (MPE)** | 探索模式 UI | 探索面板、Ghost Node、截图标注交互 |
|  | Explore Store | 维护会话状态、当前提议、审阅操作和恢复信息 |
|  | Flow 集成层 | 把确认后的步骤落地到现有画布和历史系统 |
|  | 通信协调 | 向后端发送目标、反馈与控制命令，接收提议与执行结果 |
| **后端 (LocalBridge)** | 会话管理器 | 创建、恢复、暂停、终止探索会话 |
|  | LLM 客户端 | 调用支持 vision 的模型，并输出结构化结果 |
|  | 截图与上下文管理 | 获取实时截图、缓存历史截图、组织多轮上下文 |
|  | mfw 执行层 | 调用截图、点击、滑动、输入、识别等原子能力 |
|  | 决策解析器 | 将模型输出转换为前端可渲染的统一提议结构 |

### 7.3 探索协议设计

建议沿用现有 `path + data` 的 WebSocket 通信风格，为探索模式单独定义协议：

| 路由 | 方向 | 用途 |
| --- | --- | --- |
| `/etl/explore/start` | 前端 → 后端 | 创建探索会话并提交目标 |
| `/lte/explore/session_started` | 后端 → 前端 | 返回 `session_id` 与初始状态 |
| `/etl/explore/request_next` | 前端 → 后端 | 请求基于当前画面生成下一步提议 |
| `/lte/explore/proposal` | 后端 → 前端 | 返回当前步骤提议、标注、候选方案和摘要 |
| `/etl/explore/review` | 前端 → 后端 | 提交确认、修改、拒绝、回退等审阅结果 |
| `/etl/explore/test_step` | 前端 → 后端 | 请求先测试当前提议 |
| `/lte/explore/execution_result` | 后端 → 前端 | 返回测试或正式执行结果 |
| `/etl/explore/pause` | 前端 → 后端 | 暂停当前会话 |
| `/etl/explore/resume` | 前端 → 后端 | 恢复当前会话 |
| `/etl/explore/terminate` | 前端 → 后端 | 终止当前会话 |
| `/lte/explore/error` | 后端 → 前端 | 返回可恢复或不可恢复错误 |

核心字段建议统一包含：

- `session_id`
- `step_id`
- `proposal_id`
- `controller_id`
- `screenshot_id`
- `timestamp`
- `recoverable`

### 7.4 核心数据结构建议

为避免前后端理解不一致，建议先定义统一提议结构：

```ts
type ExploreProposal = {
  session_id: string;
  step_id: string;
  proposal_id: string;
  screenshot_id: string;
  confidence: number;
  reasoning_summary: string;
  needs_human_attention: boolean;
  overlays: Array<{
    type: "roi" | "target" | "begin" | "end" | "hint";
    rect: [number, number, number, number];
    label?: string;
    score?: number;
  }>;
  node_draft: {
    label: string;
    recognition: {
      type: string;
      param: Record<string, unknown>;
    };
    action: {
      type: string;
      param: Record<string, unknown>;
    };
    others: Record<string, unknown>;
  };
  alternatives?: Array<{
    title: string;
    summary: string;
    node_draft: ExploreProposal["node_draft"];
  }>;
};
```

其中 `node_draft` 应尽量与现有 Pipeline Node 的 `recognition / action / others` 结构保持一致，降低落地成本。

### 7.5 Ghost Node 的工程落地策略

建议采用以下策略：

- **不新增持久化节点类型**，保持正式节点仍为现有 Pipeline Node
- 前端单独维护 `draftMeta`，如 `isGhost`、`proposalId`、`confidence`、`overlayBindings`
- 只有在用户确认后，才把 `node_draft` 写入现有流程 Store
- 未确认的 Ghost Node 不参与正常保存与导出

这样既能复用现有节点渲染与字段编辑能力，也能降低对当前画布模型的侵入性。

### 7.6 持久化、日志与回放

建议从一开始就为可恢复性预留结构：

- **会话快照**：保存最近一次会话状态、目标、步骤号和已确认链
- **截图缓存**：为最近若干步保留前后截图和标注信息
- **审阅日志**：记录用户对每个提议的确认、修改、拒绝和回退
- **回放能力**：后续可基于日志回放探索过程，用于调试与产品改进

MVP 可以先做“最近会话恢复”，完整回放可放后续阶段。

## 8. 与现有功能的结合

### 8.1 LocalBridge 与 mfw 能力复用

- 复用现有截图、OCR、点击、滑动、输入等能力
- 探索模式不重新发明执行层，只新增会话与决策编排层
- 执行结果继续走既有协议风格，降低集成成本

### 8.2 调试功能结合

- 每步确认前可选择“先测试”
- 测试结果可作为当前提议的补充反馈
- 后续可基于测试失败原因触发“重新提议”

### 8.3 字段面板结合

- 当前 Ghost Node 可直接复用字段面板进行参数微调
- 用户修改后的字段应回写到当前提议，并覆盖 AI 原始输出
- 现有智能补全能力继续作为字段级辅助存在

### 8.4 模板系统结合

- AI 提议可优先参考已有模板和历史节点模式
- 用户确认后的高质量步骤可沉淀为模板
- 后续可实现“基于模板的探索偏置”，降低重复性场景的决策成本

### 8.5 现有 Store 与画布逻辑结合

建议新增独立的 `Explore Store`，而不是把探索状态硬塞进现有 Flow Store：

- **Explore Store**：会话状态、提议、截图、标注、面板 UI、恢复信息
- **Flow Store**：正式节点、边、历史、视图、路径等既有编辑状态

两者的交界点只有一个：

> 当用户确认当前提议时，才把 `node_draft` 写入 Flow Store，并纳入正常历史记录。

## 9. 验收标准与指标

### 9.1 功能验收

MVP 至少应满足以下能力闭环：

1. 已连接控制器后，可以成功创建探索会话
2. 可以基于当前截图拿到结构化提议
3. 用户可以确认、修改、拒绝、回退、终止当前步骤
4. 确认后可以执行该步骤，并获得执行结果与新截图
5. 失败、断连、截图异常时，系统能给出清晰反馈并支持恢复或终止

### 9.2 体验指标

建议关注以下指标，用于衡量该模式是否真正有价值：

- **首个可运行流程完成时间**
- **单步提议确认率**
- **单步人工修改率**
- **因执行失败触发回退的比例**
- **从截图到提议返回的耗时**
- **会话中断后的恢复成功率**

### 9.3 实施里程碑建议

| 里程碑 | 内容 |
| --- | --- |
| **M1** | 定义探索协议、会话模型和前端状态结构 |
| **M2** | 打通单步提议展示、Ghost Node 和截图标注 |
| **M3** | 打通确认执行、结果回流、回退和错误处理 |
| **M4** | 增强分支处理、维护适配、模板沉淀与会话恢复 |

## 10. 价值与愿景

### 10.1 用户价值

| 用户类型 | 获得价值 |
| --- | --- |
| **新手** | 降低学习门槛，在引导下完成首个 Pipeline |
| **熟练开发者** | 提升编写效率，减少机械重复劳动 |
| **维护者** | 更快定位失效步骤，降低 UI 适配成本 |

### 10.2 长期愿景

- **个人知识沉淀**：用户的确认、修改与拒绝记录可形成个人偏好素材
- **团队知识共享**：高质量探索结果可沉淀为团队模板与最佳实践
- **生态共建**：未来可形成基于场景的探索策略库，而不是只沉淀零散节点

## 11. 相关概念区分

| 概念 | 说明 | 与本模式的关系 |
| --- | --- | --- |
| **MCP** | 模型上下文协议，强调标准化工具调用 | 本模式当前不依赖 MCP，优先复用 LocalBridge 直接调用 mfw 的能力 |
| **Skills** | 文档 + 脚本的工作流封装 | 探索结果未来可进一步沉淀为模板、策略或 Skill |
| **传统 AI 补全** | 单节点级别的字段补全 | 本模式是流程级协作，传统补全仍作为子能力存在 |
| **普通编辑模式** | 用户直接在画布上手动编排节点 | 探索模式是增强层，可随时切换回普通编辑模式 |

## 12. 待决策问题

以下问题可以在实现前进一步确认：

- MVP 默认接入哪类模型，是否优先支持本地模型
- 敏感信息场景下，是否需要截图脱敏或仅允许本地推理
- 会话恢复是只保留内存态，还是落盘到本地缓存
- “先测试”是否允许对真实设备造成轻量副作用，还是仅做纯识别验证
- 在维护适配场景中，是否需要专门的“旧节点 vs 新提议”对比视图
