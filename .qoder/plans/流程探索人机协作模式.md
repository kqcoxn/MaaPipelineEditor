# RLHF AI 辅助流程探索模式（MPE探索模式）

## 1. 背景与动机

### 1.1 当前痛点

在 MaaFramework Pipeline 的开发过程中，开发者通常会遇到以下问题：

- **古法编写效率低**：需要手动截图、逐个配置节点、反复调试验证
- **AI 辅助视野有限**：现有 AI 补全大多停留在单节点或字段级别，缺少“流程下一步该做什么”的整体判断
- **验证周期长**：往往要把一串步骤写完后再整体运行，问题发现得晚、回溯成本高
- **维护成本高**：一旦游戏 UI 改版，旧流程容易局部失效，但排查通常只能靠人工逐步试错

### 1.2 核心洞察

借鉴 RLHF（Reinforcement Learning from Human Feedback）中的"人在回路"思想，但不追求自动训练本身，而是强调协作闭环。

**核心特性：流程级辅助，节点级 RLHF**

- **流程级辅助**：Agent 具备全局视野，能够理解整个 Pipeline 的目标和上下文，提供跨节点的流程级建议，而非孤立的单节点补全
- **节点级 RLHF**：每个节点的生成、确认、修改、拒绝都构成一次人类反馈，Agent 在节点级别学习用户偏好，持续优化后续提议质量

其核心为，问题的难点不在于"让模型生成一个节点"，而在于：

- 当前画面是动态的、不完全可观测的
- 下一步动作依赖前序步骤、执行结果和用户修正
- 执行后必须校验结果，失败后要能够重规划而不是继续硬跑
- 涉及敏感动作、歧义目标、不可逆操作时必须让人保留控制权

因此，这个能力不应被定义为“提示词工程增强版”，而应被定义为一个**运行在前端、有状态、可调用工具、可被人审阅和打断的 RLHF Agent Runtime**。

> 真正需要设计的不是一条 prompt，而是一套“观测 → 记忆 → 规划 → 提议 → 审阅 → 执行 → 验证 → 重规划”的协作运行机制。

### 1.3 设计原则

- **先提议，后落地**：Agent 产出的是待确认提议，不直接写入正式流程
- **单 Agent 优先，角色分层实现**：MVP 先做一个会话型 RLHF Agent，在内部拆分感知、记忆、规划、验证等职责，而不是一开始就上多 Agent 编排
- **每步可审阅、可中断、可回退**：任何一步都应可观察、可修改、可撤销、可恢复
- **结构化优先**：Agent 输出应尽可能是稳定的数据结构，而非自由文本
- **人始终拥有最终决定权**：涉及歧义、高风险、不可逆动作时必须保留人工把关
- **Prompt 只是实现细节**：真实能力边界来自 prompt、工具契约、状态机、工作记忆、安全规则和审阅闭环的组合
- **AI 前置，执行后置**：AI 推理、会话状态、工作记忆和提议组装全部在前端完成；后端只负责 mfw 相关能力调用与结果回传
- **与现有工作流共存**：探索模式增强编辑体验，不替代现有的字段编辑、调试、模板与手工编排能力

## 2. 模式定义

### 2.1 什么是 RLHF AI 辅助流程探索模式

RLHF AI 辅助流程探索模式是一套**流程级的人机协作工作流能力**。其中的 AI 核心，不再被视为一次性的"提词器"，而是一个**运行在前端、会话范围内受约束的 RLHF Agent**。

其中，对外产品表达可简称为 **“MPE探索模式”**；本规划正式名称统一为 **"RLHF AI 辅助流程探索模式"**。需要注意的是，`RLHF Agent` 是该模式内部的智能体运行时，而不是整个模式名称。其设计哲学是：

> 在流程层面提供智能辅助决策，在节点层面通过人类反馈持续优化——即"流程级辅助，节点级 RLHF"。

| 角色                  | 职责                                                                                             |
| --------------------- | ------------------------------------------------------------------------------------------------ |
| **用户**              | 给出目标、审阅提议、修正歧义、决定是否执行、在必要时手动接管                                     |
| **RLHF Agent**        | 根据当前画面、历史上下文、已确认步骤和用户反馈，决定“下一步该做什么”，并输出结构化提议与备选方案 |
| **MPE 前端**          | 承载探索会话、工作记忆、模型调用、截图标注、Ghost Node 预览、提议审阅和流程落地                  |
| **LocalBridge / mfw** | 仅提供截图、识别、点击、滑动、输入、等待、调试执行等底层能力，并将确认后的步骤真正执行到设备上   |

### 2.2 为什么这是类 Agent 架构，而不是提示词工程

如果把这个问题当成“截图 + prompt + JSON 输出”，会天然漏掉以下核心能力：

- **会话状态**：同一个目标要跨越多轮截图、多个步骤和多次人工修正
- **工具使用**：模型输出不是终点，必须联动截图、识别、点击、测试、执行等工具
- **短程规划**：需要在当前上下文下决定主路径、备选路径、验证条件和终止条件
- **反馈吸收**：用户修改 ROI、否决提议、手动插入步骤后，后续决策必须受影响
- **失败恢复**：执行结果不符合预期时，需要进入重规划而不是继续沿原路径前进
- **安全守卫**：敏感动作、高风险动作、不可逆动作必须受到规则限制

因此，这里的“agent”不是指一个开放式、无限自治的系统，而是指一个**受约束、短视距、强审阅的人机协作代理**。

### 2.3 RLHF Agent 的内部组成

推荐将 RLHF Agent 设计为一个单运行时、内部多职责模块的结构：

| 模块                         | 职责                                                                     |
| ---------------------------- | ------------------------------------------------------------------------ |
| **Perception / Observation** | 获取截图、OCR、目标候选区域、控件上下文等观测信息                        |
| **Working Memory**           | 保存目标、当前步骤、已确认链路、用户修正、失败记录、安全标记等会话内记忆 |
| **Planner / Policy**         | 根据当前观测和工作记忆生成短程计划，决定当前主提议与有限备选             |
| **Proposal Assembler**       | 将决策结果转换为前端可审阅的结构化提议、Ghost Node 和截图标注            |
| **Review Adapter**           | 吸收用户的确认、修改、拒绝、回退、手动接管等反馈，回写工作记忆           |
| **Execution Supervisor**     | 区分测试与正式执行，调用 mfw 原子能力并收集执行回流                      |
| **Verifier / Critic**        | 判断执行结果是否符合预期，失败时触发重规划或人工介入                     |
| **Safety Guard**             | 检查敏感动作、不可逆动作、低置信度、高歧义等情况，决定是否必须人工确认   |
| **Trace Recorder**           | 记录观测、提议、审阅、执行、验证和恢复信息，用于回放、调试和模板沉淀     |

### 2.4 核心交互闭环

```text
用户设定目标
      ↓
Observe：LocalBridge 获取当前画面与上下文
      ↓
Update Memory：写入工作记忆（目标、历史、用户修正、失败信息）
      ↓
Plan：RLHF Agent 进行短程规划，产出主提议和有限备选
      ↓
Propose：MPE 呈现 Ghost Node + 截图标注 + 决策摘要
      ↓
Review：人审阅并操作 [确认] [修改] [拒绝] [测试] [回退] [接管] [终止]
      ↓
Execute / Test：通过 LocalBridge / mfw 执行或验证当前步骤
      ↓
Verify：判断结果是否符合预期
      ↓
成功 → 写入 confirmed_chain → 下一轮探索
失败 → 重规划 / 人工介入 / 终止
```

### 2.5 范围边界

本模式：

- 不是“全自动无人值守脚本生成器”
- 不是对普通编辑模式的替代，而是其增强层
- 不是单节点 AI 补全的简单放大版，而是带有会话、状态、工具调用和执行闭环的流程协作模式
- 不是一开始就做“多 Agent 自组织系统”，MVP 先做单 Agent Runtime，把工程可控性放在第一位

## 3. 目标用户与典型场景

### 3.1 场景一：新手快速上手

**用户**：刚接触 MaaFramework 的新开发者  
**目标**：完成游戏登录流程

**流程**：

1. 用户在 MPE 中输入目标：“完成游戏登录”
2. RLHF Agent 分析当前游戏画面，提议第一步：“点击账号输入框”
3. MPE 在画布上显示半透明 Ghost Node，并在截图中高亮推荐区域
4. 用户确认位置正确，或微调 ROI 后点击“确认”
5. 节点变为正式节点，LocalBridge 执行该步骤
6. 执行结果回流后，RLHF Agent 更新工作记忆，继续推进入下一步

**价值**：新手无需先理解 MaaFramework 的字段细节，也能在引导下完成首个可运行 Pipeline。

### 3.2 场景二：复杂流程拆解

**用户**：有经验的开发者  
**目标**：完成游戏内一个多步骤活动

**流程**：

1. 用户输入目标：“完成每日签到活动”
2. RLHF Agent 识别出关键路径：点击活动入口 → 切换签到标签 → 点击签到按钮 → 关闭奖励弹窗
3. 每一步都在 MPE 中以可审阅提议呈现，用户可根据实际情况微调
4. 遇到分支时，例如“今日已签到”或“活动入口位置变化”，Agent 给出主提议和有限备选
5. 执行偏离预期时，系统进入验证失败和重规划，而不是静默继续

**价值**：把复杂流程中的“探索脑力”转移给 Agent，人主要负责异常判断与结果把关。

### 3.3 场景三：流程维护与适配（Phase 2 增强）

> **说明**：此场景为探索模式的增强能力，非 Agent 核心功能。Agent 的核心能力是"基于当前画面生成新节点提议"，维护场景只是在此基础上增加"已有节点对比"的辅助信息。

**用户**：需要维护已有 Pipeline 的开发者  
**目标**：适配游戏 UI 改版

**流程**：

1. 用户加载已有 Pipeline，启动探索模式
2. （可选）系统提供当前步骤对应的已有节点信息作为 Agent 参考
3. RLHF Agent 基于当前画面生成新节点提议
4. 在维护场景下，可展示已有节点与新提议的差异摘要，辅助用户判断
5. 用户逐条审阅并确认新的适配方案

**价值**：降低 UI 改版后的维护成本，帮助用户更快定位失效点并完成修复。

### 3.4 场景四：人机接力编辑

**用户**：熟悉编辑器但不想被 AI 绑住流程的开发者  
**目标**：在关键步骤由 AI 提议，在细节步骤手工接管

**流程**：

1. 用户让 RLHF Agent 先完成主路径探索
2. 在某个复杂节点处切换到普通编辑模式，手动调整参数
3. 保存后继续探索，Agent 将手动编辑结果视为新的权威上下文

**价值**：探索模式不会“锁死”用户，而是可以自然退回现有编辑工作流。

## 4. 产品目标与阶段范围

### 4.1 产品目标

- 降低从“有目标”到“拿到首个可运行流程”的门槛
- 缩短流程编写与验证之间的反馈回路
- 降低复杂流程和维护场景中的认知负担
- 把 AI 能力从字段补全提升到流程级协作 Agent

### 4.2 MVP 范围

MVP 聚焦最小可用闭环，核心不是“做一个很会说的 prompt”，而是“做一个可控的前端会话型 Agent Runtime”：

- 单控制器、单活跃探索会话
- 单个前端 RLHF Agent Runtime
- 前端复用现有 AI 配置、`OpenAIChat` 会话封装和 AI History 能力
- 后端仅复用现有 `mfwProtocol` / `DebugProtocol` 的执行与结果回传能力
- 短视距规划：每轮产出 **1 个主提议 + 最多 2 个备选**
- 单步提议 → 人确认或修改 → 测试 / 执行 → 验证 → 下一步 / 重规划
- 支持基础动作类型：点击、滑动、输入、等待、判断、结束、请求人工介入
- 支持截图标注、Ghost Node 预览、决策摘要展示
- 支持右下角 AI 入口、大尺寸协作 Modal、对话时间线、当前提议卡和底部操作栏
- 支持回退一步、拒绝当前提议、手动接管、终止探索
- 支持最近会话恢复
- 支持结构化提议契约、工作记忆和基础安全规则

### 4.3 暂不纳入 MVP 的内容

- 多控制器并行探索
- 多 Agent 协作编排
- 长链路全自动无人值守生成
- 大规模并行分支树生成
- 基于用户历史自动训练个人偏好模型

### 4.4 阶段演进建议

| 阶段                | 目标                  | 核心能力                                                       |
| ------------------- | --------------------- | -------------------------------------------------------------- |
| **Phase 1**         | 打通单 Agent 探索闭环 | 会话管理、工作记忆、单步提议、Ghost Node、确认执行、验证与回退 |
| **Phase 2**         | 提升复杂场景可用性    | 分支候选、失败重规划、维护适配（已有节点对比）、模板辅助       |
| **Phase 3**         | 形成知识沉淀能力      | 会话回放、模板沉淀、偏好记忆、团队共享                         |
| **Phase 4（可选）** | 评估更强 agentic 能力 | Planner/Critic 拆分、多阶段策略、有限多 Agent 协作             |

说明：

- **多 Agent 不是 MVP 的必要条件**
- 先把单 Agent Runtime 做稳，再决定是否拆出 Planner / Critic 等子代理

## 5. 交互与状态设计

### 5.1 探索会话模型

探索模式应被建模为一个独立的 **Explore Session**，其核心不是“当前 prompt 内容”，而是“当前 agent 在这个会话里的状态”。

| 状态项                  | 说明                                                                                   |
| ----------------------- | -------------------------------------------------------------------------------------- |
| **session_id**          | 当前探索会话唯一标识                                                                   |
| **controller_id**       | 当前关联的设备或控制器                                                                 |
| **goal**                | 当前探索目标描述                                                                       |
| **status**              | 会话生命周期状态，例如 active / paused / completed / failed                            |
| **agent_state**         | Agent 当前内部阶段，如 observing / planning / awaiting_review / verifying / replanning |
| **current_step**        | 当前正在处理的第 N 步                                                                  |
| **working_memory**      | 会话内工作记忆，包含已确认链、用户修正、失败信息、约束条件                             |
| **current_plan**        | 当前短程计划，包括主提议、备选、预期结果和验证条件                                     |
| **proposal**            | 当前待审阅的结构化提议                                                                 |
| **screenshot_context**  | 当前步骤关联的截图、ROI、标注信息                                                      |
| **confirmed_chain**     | 已确认并落地的步骤序列                                                                 |
| **review_history**      | 用户确认、修改、拒绝、回退、接管等操作记录                                             |
| **verification_result** | 最近一次执行 / 测试后的验证结果                                                        |
| **safety_flags**        | 当前是否存在敏感、高风险、不可逆等安全标记                                             |
| **last_error**          | 最近一次失败信息或警告                                                                 |

额外约束：

- 同一时刻一个控制器只允许存在一个活跃探索会话
- 会话状态应独立于普通画布编辑状态，避免污染既有文件编辑逻辑
- 用户手动修改后的结果必须高优先级写入 `working_memory`

### 5.2 探索状态机

需要显式定义探索模式的状态迁移，避免“UI 看起来能点，但后台语义不清”的情况：

| 状态                   | 含义                             | 允许迁移                                               |
| ---------------------- | -------------------------------- | ------------------------------------------------------ |
| **idle**               | 尚未开始探索                     | observing / terminated                                 |
| **observing**          | 正在获取截图与上下文             | planning / failed                                      |
| **planning**           | 正在基于工作记忆生成短程计划     | reviewing_proposal / failed                            |
| **reviewing_proposal** | 用户正在审阅当前提议             | testing / executing / replanning / paused / terminated |
| **testing**            | 用户选择先测试当前步骤           | verifying / failed                                     |
| **executing**          | 已确认并正在正式执行步骤         | verifying / failed                                     |
| **verifying**          | 正在判断结果是否符合预期         | observing / replanning / completed / failed            |
| **replanning**         | 因失败、拒绝或环境变化而重新规划 | reviewing_proposal / paused / failed                   |
| **paused**             | 用户手动接管或暂时中止           | observing / planning / terminated                      |
| **failed**             | 当前步骤或会话失败               | replanning / paused / terminated                       |
| **completed**          | 目标达成，探索结束               | idle                                                   |
| **terminated**         | 用户主动终止                     | idle                                                   |

补充说明：

- 页面刷新或 WebSocket 重连后，若检测到未结束会话，可提供“恢复最近会话”入口
- 一旦控制器断连、截图失败、执行层不可用，会话应自动进入 `failed` 或 `paused`
- `replanning` 是 agent 架构的重要状态，不能被简化成再次请求同一个 prompt

### 5.3 Working Memory 建模

工作记忆是 RLHF Agent 区别于“单轮提示词”的关键。建议至少拆分为以下几类：

| 记忆类型                   | 内容                                             |
| -------------------------- | ------------------------------------------------ |
| **Goal Memory**            | 用户目标、范围限制、完成条件                     |
| **Interaction Memory**     | 已确认步骤、当前已进入的页面、关键上下文变量     |
| **User Correction Memory** | 用户手动调整的 ROI、参数、命名、分支选择偏好     |
| **Failure Memory**         | 最近失败的动作、失败原因、禁止重复尝试的错误路径 |
| **Safety Memory**          | 敏感页面、不可逆动作、必须二次确认的情形         |
| **Template Hint Memory**   | 来自模板、历史节点、现有 Pipeline 的偏置提示     |

记忆写入规则建议：

- 用户明确修改过的内容优先级高于模型原始判断
- 执行失败但未确认原因的信息只作为短期记忆，不应直接沉淀为模板
- 只有“确认 + 验证通过”的高质量步骤，才有资格进入长期沉淀

### 5.4 Ghost Node（幽灵节点）

Ghost Node 是 Agent 当前提议的可视化承载物，推荐采用**“现有 Pipeline Node + Draft 元信息”**的方式落地，而不是直接新增一种可持久化节点类型。

原因：

- 现有画布节点体系已经稳定，正式节点的数据结构也已明确
- Ghost Node 本质上是“待确认渲染态”，不是最终存档态
- 未确认提议不应进入正式 Pipeline JSON，也不应污染正常的导出与保存逻辑

推荐表现：

- 半透明或降低饱和度
- 虚线边框，显式区分“待确认”
- 显示置信度、动作类型、关键识别信息、风险级别
- 支持拖拽、框选调整、双击编辑字段
- 提供显式操作：确认、修改、拒绝、测试

### 5.5 截图联动标注

Agent 的提议必须与截图强绑定，至少包含：

- 当前 `screenshot_id`
- Agent 推荐的 ROI / target / begin / end 区域
- 候选目标框、置信度和标签
- 用户人工修正后的最终区域
- 预期执行结果的验证区域或验证条件

交互要求：

- 用户可拖动调整框选范围
- 用户可重新框选并把结果回灌给 Agent，或直接覆盖当前提议
- 一旦用户手动修改 ROI，该修改应视为当前步骤的权威输入

### 5.6 探索入口与唤起机制

建议将 RLHF AI 辅助流程探索模式（MPE探索模式）的主入口设计为**右下角悬浮 AI 圆形按钮（FAB）**，而不是常驻占用顶部工具栏或左侧主工具区。

原因：

- 探索模式是按需进入的协作态，不应打断用户的常规手工编排
- 右下角更符合“助手 / 会话入口”的心理预期，也便于与画布主操作区分层
- 能兼容当前已有顶部工具栏、左上全局工具条和 Modal 型截图/识别工具的交互习惯

#### 5.6.1 与现有面板的位置关系

FAB 的定位需与现有界面元素协调，避免重叠冲突：

| 现有面板                                      | 位置                               | z-index | 与 FAB 的关系                                    |
| --------------------------------------------- | ---------------------------------- | ------- | ------------------------------------------------ |
| **ToolbarPanel**（顶部工具栏）                | `top: 10px, right: 10px`           | 30      | 位于右上角，FAB 位于右下角，垂直方向不重叠       |
| **SearchPanel**（搜索面板）                   | `top: 70px, right: 10px`           | 10      | 位于右侧中上区域，FAB 位于右下角，需保持垂直间距 |
| **FieldPanel / EdgePanel**（字段/边属性面板） | `right: 10px, top: 70px`（可拖动） | 20      | 右侧浮动面板，FAB 需预留底部空间避免遮挡         |
| **JsonViewer**（JSON 查看器）                 | 右侧浮动                           | -       | 右侧面板，与 FAB 位置错开                        |
| **ConfigPanel / AIHistoryPanel**              | `left: 6px, top: 6px`              | 20      | 左上角，与 FAB 位置完全错开                      |
| **GlobalPanel / AddPanel**（工具面板）        | `left: 10px, top: 10px/80px`       | 10      | 左侧工具条，与 FAB 位置完全错开                  |

**FAB 推荐定位**：

```css
.explore-fab {
  position: absolute;
  right: 16px;
  bottom: 16px;
  z-index: 50; /* 高于现有面板，低于 Modal 遮罩层 */
  width: 56px;
  height: 56px;
  border-radius: 50%;
}
```

**边界约束**：

- FAB 与右侧浮动面板（SearchPanel、FieldPanel 等）保持至少 `16px` 的水平/垂直间距
- 当右侧浮动面板展开时，FAB 不应被遮挡；若面板尺寸动态变化，应考虑 FAB 的自动避让或固定优先级
- FAB 的 z-index 设为 `50`，高于现有工具面板（z-index: 10~30），但低于协作 Modal 遮罩层（z-index: 100）
- FAB 始终相对于 `.workspace` 定位，随画布缩放/平移保持固定位置

入口状态建议：

- **idle**：默认机器人图标 + `MPE探索模式` 提示，点击后弹出“新建探索 / 恢复最近会话”
- **active**：按钮外圈展示当前阶段色环，角标显示当前步骤号或进度
- **awaiting_review**：高亮或未读点提示，表示 Agent 已给出新提议，等待用户审阅
- **paused / failed**：灰红态，点击优先进入恢复会话或查看错误原因
- **disconnected**：禁用态，点击提示“请先连接本地服务与设备”

入口交互建议：

- **单击**：打开或聚焦协作 Modal
- **长按 / 右键**：显示快捷操作，如“新建会话”“恢复最近会话”“终止当前会话”
- 关闭 Modal 不等于结束会话；若存在活跃会话，FAB 应继续承载状态提示
- 用户从探索模式切回普通编辑后，FAB 仍应作为“返回协作态”的锚点

### 5.7 协作 Modal 布局与 UI 风格

点击 FAB 后，建议打开一个**占据大部分屏幕的居中 Modal 工作台**，而不是右侧停靠式侧边栏。推荐尺寸为：

- 宽度：`min(1440px, calc(100vw - 48px))`
- 高度：`calc(100vh - 48px)`
- 位置：居中浮层，保留外层画布作为上下文背景
- 结构：Modal 内部再做多栏布局，而不是再套一层抽屉

#### 5.7.1 Modal 与现有界面的层级关系

协作 Modal 作为全屏工作台，其层级设计需明确与现有元素的遮罩关系：

```
┌─────────────────────────────────────────────────────────────┐
│ Header（全局顶部，不受 Modal 影响）                          │
├─────────────────────────────────────────────────────────────┤
│ workspace（工作区容器）                                       │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 底层画布 + 现有面板（被 Modal 遮罩覆盖，不参与交互）        │ │
│  │  - GlobalPanel/AddPanel/LayoutPanel/DebugPanel (左侧)   │ │
│  │  - ToolbarPanel/SearchPanel (右侧上部)                  │ │
│  │  - ConfigPanel/AIHistoryPanel (左上浮动)                │ │
│  │  - FieldPanel/EdgePanel/JsonViewer (右侧浮动)           │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ Modal 遮罩层 (z-index: 100)                             │ │
│  │  - 半透明背景，隔离底层交互                              │ │
│  │  - 可选：保留画布内容作为背景上下文                      │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 协作 Modal 内容层 (z-index: 101)                        │ │
│  │  - 居中工作台：左观测区 + 中时间线区 + 右草稿区           │ │
│  │  - 底部操作栏                                           │ │
│  │  - 头部会话条                                           │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ FAB（探索入口，Modal 打开后可隐藏或变为最小化按钮）       │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

**层级与 z-index 规范**：

| 层级 | 元素                    | z-index | 说明                                               |
| ---- | ----------------------- | ------- | -------------------------------------------------- |
| 0    | 画布底图                | 0       | ReactFlow 主画布                                   |
| 1    | 左侧工具面板            | 10      | GlobalPanel、AddPanel、LayoutPanel、DebugPanel     |
| 2    | 常规浮动面板            | 20      | ConfigPanel、AIHistoryPanel、FieldPanel、EdgePanel |
| 3    | 顶部工具栏              | 30      | ToolbarPanel（最高优先级的常驻 UI）                |
| 4    | **FAB 探索入口**        | **50**  | 高于所有常驻面板，随时可见                         |
| 5    | **Modal 遮罩层**        | **100** | 隔离底层交互                                       |
| 6    | **协作 Modal 内容**     | **101** | 最高层级，全屏工作台                               |
| 7    | NodeAddPanel 等临时弹窗 | 101+    | 与协作 Modal 平级或更高                            |

**遮罩行为**：

- Modal 打开时，所有底层面板（包括 FAB）的交互被遮罩层阻断
- 可选策略：FAB 在 Modal 打开后变为「最小化」按钮，固定在 Modal 头部或右下角，用于最小化 Modal 返回画布视图
- Modal 关闭后，恢复所有底层面板的交互能力
- Ghost Node 在 Modal 关闭但仍存在活跃会话时，应在画布上保留显示（作为待确认状态的视觉提示）

**与 Header 的关系**：

- Header（包含版本信息、连接状态、主题切换等）始终可见，不被 Modal 遮罩
- Modal 的 `calc(100vh - 48px)` 高度已预留 Header 空间
- Header 中的设备连接状态应与探索会话状态联动显示

建议采用**左观测区 + 中时间线区 + 右草稿区 + 底部操作栏**的工作台布局：

- **头部会话条**：显示目标摘要、控制器、当前阶段、步骤进度、连接状态，以及“最小化 / 暂停 / 终止 / 关闭”操作
- **左侧观测区（约 45% ~ 50%）**：主截图画布、Overlay 标注、前后截图对比、缩放与重取截图工具
- **中部时间线区（约 28% ~ 32%）**：展示目标输入、Agent 观察摘要、主提议、备选方案、执行结果、验证结论和错误恢复消息
- **右侧草稿区（约 22% ~ 26%）**：通过 Tabs 承载 `当前提议`、`草稿字段`、`已确认步骤`、`差异对比`、`工作记忆摘要`
- **底部操作栏**：统一承载自然语言输入框与主操作按钮（确认、测试、修改、拒绝、回退、接管、终止）

在视觉风格上，应尽量与现有其他组件保持一致，而不是额外创造一套“AI 专属皮肤”：

- **基础风格**：延续现有 Ant Design + 白底浅边框 + 轻阴影的产品语言，避免做成黑客风、悬浮聊天窗或强拟物卡片
- **色彩与层次**：主色继续使用现有系统蓝（与 Tabs/Primary Button 一致），背景以白色、浅灰色块和 `#e8e8e8` 边框为主，风险/失败/成功分别用橙、红、绿辅助
- **圆角与间距**：沿用现有 Modal/面板的 `8px ~ 12px` 圆角和 `12px / 16px` 间距节奏，信息卡片尽量规整、克制
- **Tabs 风格**：复用现有 `antd.less` 中的无边框 Tabs + 底部蓝色激活线，不做新的胶囊导航体系
- **信息卡片**：当前提议卡、验证结果卡、差异卡均使用现有面板式白底卡片，顶部标题 + 状态标签 + 内容摘要结构即可
- **Overlay 风格**：截图框选采用蓝色主框、橙色风险框、绿色验证通过框、红色失败框，避免同时出现过多颜色
- **动效**：只保留淡入、缩放、Tabs 切换等 150~200ms 的轻动效，强调稳定和工具感，而不是聊天产品化动效

推荐默认焦点：

- 新建会话时，默认聚焦“目标输入框”
- Agent 返回提议后，自动切到 `当前提议`
- 用户点击截图标注或 Ghost Node 后，自动切到 `当前提议` 或 `草稿字段`
- 当窗口宽度不足以稳定容纳三栏时，优先保留左观测区与中时间线区，右侧草稿区折叠为 Modal 内 Tabs 次级视图，而不是退化为外部侧边栏

### 5.8 对话流与画布 / 截图联动

探索模式中的“对话框”不应只是聊天容器，而应是整个会话的**可回放时间线**。建议至少支持以下消息类型：

- **用户消息**：目标补充、纠正、分支选择、自然语言修改指令
- **Agent 消息**：观察摘要、当前提议、备选方案、风险说明
- **系统消息**：执行中、验证通过 / 失败、断连、恢复成功
- **工具消息**：截图、识别、点击、等待等关键执行回流

联动规则建议：

- 点击某条提议消息时，画布自动高亮对应 Ghost Node，并在截图区定位相关 overlay
- 点击 Ghost Node 或截图框选时，对话时间线自动滚动到对应提议记录
- 用户在截图上重新框选后，应自动生成一条“用户修正”消息并写回 `working_memory`
- 执行结果应以新卡片追加到时间线，而不是覆盖掉原提议，保证“提议 → 执行 → 验证”链路可追溯

### 5.9 回退、接管与恢复

探索模式不仅要支持“向前走”，也必须明确“如何安全撤回”：

- **确认**：把 Ghost Node 正式写入当前流程，并记录到会话历史和工作记忆
- **回退一步**：撤销最近一次确认，同时回滚前端会话上下文与最近一次草稿提交
- **拒绝当前提议**：保留当前步骤号，但要求 Agent 基于同一画面和拒绝原因重新给出新方案
- **手动接管**：临时切回普通编辑逻辑，用户修改后再继续探索
- **恢复会话**：断连或刷新后，允许从前端快照恢复最近会话的目标、步骤、工作记忆和历史截图

### 5.10 分支处理

MVP 阶段建议严格控制复杂度：

- 默认只展示 **1 个主提议 + 最多 2 个备选提议**
- 备选提议需要明确其触发前提，而不是只列出不同动作
- 用户先在备选中做选择，再继续后续探索
- “一次生成完整分支树”放到后续阶段，而不是 MVP 强行覆盖

## 6. RLHF Agent 决策与执行模型

### 6.1 前端 Agent Loop

建议将核心运行过程显式建模为**前端驱动的 Agent Loop**：

| 阶段                  | 前端责任模块        | 依赖能力                        | 输出                                |
| --------------------- | ------------------- | ------------------------------- | ----------------------------------- |
| **Observe**           | Observation Service | `mfwProtocol` 截图 / OCR        | 标准化观测结果 `observation_bundle` |
| **Compose Context**   | Context Composer    | 工作记忆 + 当前画面 + 模板      | 本轮模型输入 `agent_input`          |
| **Plan**              | Planning Core       | `OpenAIChat` / AI Provider      | 原始提议 JSON、有限备选、验证计划   |
| **Normalize**         | Proposal Normalizer | Schema 校验 + 默认值补全        | 前端可渲染的 `ExploreProposal`      |
| **Review Align**      | Review Reducer      | 用户确认 / 修改 / 拒绝 / 接管   | 更新后的工作记忆、下一步决策        |
| **Execute / Test**    | Execution Broker    | `mfwProtocol` / `DebugProtocol` | 执行回流、前后截图、异常信息        |
| **Verify**            | Verifier / Critic   | 规则校验 + 必要时模型复核       | 成功 / 失败 / 不确定                |
| **Replan / Complete** | Session Controller  | 状态机 + Failure Memory         | 下一轮探索、人工介入或结束          |

这里的关键点是：

- 模型调用、工作记忆、提议生成、反馈吸收都发生在前端
- LocalBridge 只负责提供观测与执行工具，不负责策略判断
- 每一轮都先产出“可审阅提议”，再决定是否执行

### 6.2 智能体前端分层

为避免把智能体逻辑写成一个巨大组件，建议在前端拆为如下层次：

| 层级         | 模块                                   | 主要职责                                                               |
| ------------ | -------------------------------------- | ---------------------------------------------------------------------- |
| **会话层**   | `Explore Store` + `Session Controller` | 管理会话生命周期、状态机、当前步骤、Modal UI 状态、恢复快照            |
| **观测层**   | `Observation Service`                  | 发起截图、OCR、候选框提取、读取控制器状态，组装观测包                  |
| **上下文层** | `Context Composer`                     | 将目标、当前画面、已确认链、用户修正、模板提示、风险约束拼成模型上下文 |
| **规划层**   | `Planning Core`                        | 调用模型，要求输出严格结构化的下一步提议，而不是聊天文本               |
| **规范化层** | `Proposal Normalizer`                  | 解析 JSON、校验字段、补默认值、把结果映射到现有 Pipeline Node 结构     |
| **审阅层**   | `Review Reducer`                       | 接收用户确认、修改、拒绝、回退、接管操作，并写回工作记忆               |
| **执行层**   | `Execution Broker`                     | 选择使用直接 mfw 原子操作，或通过调试能力执行临时步骤 / 临时链路       |
| **验证层**   | `Verifier / Critic`                    | 用规则 + 观测结果判断是否命中预期，并决定重规划还是完成                |
| **追踪层**   | `Trace Recorder`                       | 记录时间线、关键截图、提议日志、验证结果、恢复快照                     |

建议实现时遵循以下边界：

- `Planning Core` 只负责“提出方案”，不直接改 Flow Store
- `Proposal Normalizer` 是模型输出进入系统的唯一入口，所有字段必须经过它
- `Execution Broker` 只执行已经过人审阅的步骤，禁止绕开审阅闭环

### 6.3 上下文拼装策略

前端在发起模型调用前，应显式构造一个**受限上下文包**，而不是把所有历史无界塞给模型。建议至少拆成以下 6 类：

| 上下文片段             | 内容                                                        |
| ---------------------- | ----------------------------------------------------------- |
| **Goal Packet**        | 当前目标、范围限制、完成条件、用户本轮补充说明              |
| **Observation Packet** | 当前截图摘要、OCR 结果、候选区域、设备状态、页面状态摘要    |
| **Chain Packet**       | 最近若干个已确认步骤、当前所处页面假设、关键分支上下文      |
| **Feedback Packet**    | 最近几次用户修改、拒绝原因、手动接管结果、命名偏好          |
| **Safety Packet**      | 敏感动作约束、禁止触达区域、必须人工确认的情形              |
| **Schema Packet**      | 当前允许的 `proposal_kind`、必填字段、`node_draft` 结构契约 |

上下文拼装规则建议：

- 用户修正优先级最高，高于模板提示和模型先验
- 历史链路只保留“本轮决策真正相关”的最近步骤，避免长上下文污染
- 维护适配场景下（可选），可将"已有节点摘要 + 当前观测差异"同时送入上下文作为参考
- 当前轮无法确定的信息，不应伪装成确定事实，而应显式标记为 `unknown`

### 6.4 规划策略

RLHF Agent 的规划建议遵循以下原则：

- **短视距规划**：默认只规划当前一步和有限备选，不追求一次性生成整条长链路
- **显式预期结果**：每个提议除了“做什么”，还要回答“做完后应该看到什么”
- **失败可重规划**：验证不匹配时，优先重新规划，而不是继续沿旧路径推进
- **人工优先兜底**：不确定时，优先输出 `ask_human`，而不是硬猜
- **偏好可继承**：若用户刚刚修改过 ROI、命名或分支选择，下一轮应默认继承

Agent 的动作意图建议统一收敛到有限类别：

- `act`
- `wait`
- `assert`
- `branch`
- `finish`
- `ask_human`

每个主提议至少应同时包含：

- 当前动作摘要
- 为什么是这一步
- 预期结果
- 验证方式
- 失败后首选回退路径
- 最多 2 个触发条件明确的备选方案

### 6.5 输出契约与前端规范化

为保证稳定性，模型输出应尽量是**结构化 JSON**，而不是自由文本。至少应包含：

- `proposal_id`
- `step_index`
- `proposal_kind`
- `node_draft`
- `overlays`
- `confidence`
- `risk_level`
- `reasoning_summary`
- `expected_outcome`
- `verification_plan`
- `alternatives`
- `needs_human_attention`

前端 `Proposal Normalizer` 需要承担以下职责：

- 移除 Markdown 外壳、代码块标记和非 JSON 噪声
- 校验必填字段是否存在
- 将模型返回字段映射为现有 Pipeline Node 的 `recognition / action / others`
- 自动补齐缺失的 UI 元信息，如 `proposalId`、`overlayBindings`、`diffSummary`
- 对不合法字段进行降级处理；若无法安全修复，则把提议降级为 `ask_human`

其中：

- `node_draft` 必须尽量贴合现有 Pipeline 节点结构
- `reasoning_summary` 只保留对用户有帮助的简短摘要，不暴露冗长内部推理
- `alternatives` 仅保留有限个候选方案，避免 UI 过载
- `verification_plan` 必须可执行；没有验证计划的提议不得直接进入执行阶段

### 6.6 用户反馈写回机制

用户反馈不应只影响当前 UI，而应回写到 Agent 的工作记忆和策略上下文：

| 用户动作     | 写回内容                                                  | 系统反应                                       |
| ------------ | --------------------------------------------------------- | ---------------------------------------------- |
| **确认**     | 将当前提议写入 `confirmed_chain`，标记为已采用方案        | 允许执行，并把本轮提议记为后续上下文依据       |
| **修改**     | 以用户修改结果覆盖当前提议，写入 `User Correction Memory` | 以后同类界面优先继承该修正                     |
| **拒绝**     | 记录拒绝原因和被否决的候选特征                            | 避免在同一画面重复产出近似低质量方案           |
| **手动接管** | 把用户手工编辑结果记为当前步骤权威结果                    | 后续规划从该手工结果继续，而不是回到模型原提议 |
| **回退**     | 回滚最近一次记忆写入和当前步骤上下文                      | 回到上一个稳定状态，等待重新提议               |
| **测试**     | 写入“当前提议待验证”状态                                  | 优先进行低副作用验证，不直接写正式链路         |

### 6.7 置信度、安全与人工介入规则

建议按置信度分层：

- **高置信度**：可以高亮推荐“确认”，但仍保留人工最终确认
- **中置信度**：默认引导用户检查 ROI、关键参数和验证条件
- **低置信度**：默认推荐人工重框选、修改或切换手动编辑

以下场景应直接要求人工介入，而不是继续自动推进：

- 屏幕元素识别冲突严重
- 动作目标不唯一
- 执行后画面变化与预期不一致
- 当前步骤涉及账号、密码、支付、删除、退出登录等敏感动作
- 当前 Agent 无法给出明确的预期结果和验证方式
- 连续两到三轮都在相近位置失败，说明已陷入局部错误路径

为了避免“AI 看起来聪明，但实际越界”：

- 涉及**不可逆动作**时必须显式二次确认
- 涉及**敏感输入**时应优先手动接管，或在用户明确授权后才执行
- 当 Agent 无法解释为何选择某一步时，不应允许其默默执行
- 当设备状态异常、控制器断连、截图失败、执行层报错时，会话必须中断并提示用户

### 6.8 测试、执行与验证策略

建议将“测试”“正式执行”“结果验证”拆成三个独立阶段：

- **识别级测试**：只验证 ROI、OCR、模板或候选区域是否成立，适合低成本快速确认
- **动作级测试**：通过现有 mfw 原子操作执行一步轻量动作，验证目标是否可达
- **调试级测试**：在需要完整节点语义时，前端临时组装 `pipeline_override`，复用 `DebugProtocol` 做单步或短链路验证
- **正式执行**：在用户确认后，真正对设备发送动作
- **结果验证**：根据 `expected_outcome` 和 `verification_plan` 判断是否成功

默认优先顺序固定为：**纯识别验证 → 低风险 direct mfw 原子动作 → `pipeline_override` 调试验证**。

验证阶段的判断顺序建议为：

1. 先走规则化验证，如截图变化、OCR 命中、状态断言
2. 规则不足时，再让模型做轻量复核，而不是默认每次都再问模型
3. 若仍不确定，落到人工确认

验证失败后不应直接报错结束，而应优先：

1. 写入 `Failure Memory`
2. 标记失败原因类别（识别失败 / 动作失败 / 验证失败 / 环境异常）
3. 进入 `replanning`
4. 给出新的主提议或请求人工介入

### 6.9 失败恢复与重规划

建议把失败恢复显式分为四类：

| 失败类型     | 典型表现                         | 默认处理                                    |
| ------------ | -------------------------------- | ------------------------------------------- |
| **观测失败** | 截图失败、OCR 为空、控制器断连   | 重试一次，仍失败则进入 `paused` 或 `failed` |
| **提议失败** | 模型输出不合法、无法映射节点结构 | 降级为 `ask_human`，并记录输出错误          |
| **执行失败** | 点击 / 输入 / 调试执行报错       | 保留现场截图，允许回退或重试                |
| **验证失败** | 执行成功但画面未进入预期状态     | 写入 `Failure Memory` 并触发 `replanning`   |

重规划时建议默认携带以下信息：

- 最近失败提议摘要
- 失败前后截图差异
- 用户是否修改过 ROI 或候选目标
- 当前禁止再次尝试的路径

### 6.10 学习与沉淀策略

MVP 不做在线训练，但应为后续沉淀预留结构：

- 记录用户确认、修改、拒绝、回退的数据
- 记录哪些提议最终通过验证，哪些提议最终失败
- 将高质量的“确认 + 验证通过”步骤沉淀为模板候选
- 为后续模板偏置、偏好记忆和维护修复建议提供数据基础

## 7. 技术架构设计

### 7.1 总体原则

**决策**：AI 与智能体能力全部前置到 MPE 前端，复用现在前端已有的 AI infra；LocalBridge 后端只负责 mfw 相关能力调用与结果回传，不承载 Agent Runtime。

原因：

- 探索会话的核心是“观测 → 规划 → 提议 → 审阅 → 执行 → 验证”的前端交互闭环
- 现有前端已经具备 AI 配置、模型调用封装与 AI 历史记录能力，扩展成本更低
- Ghost Node、时间线、Modal 工作台、字段编辑等都天然属于前端状态
- 后端保持“确定性工具层”更利于审计、复用和问题定位
- 这样能避免新增一套重型 ExploreService，把复杂度压在更容易快速迭代的前端

更准确的表述应是：

> Agent 决策是前端受约束的非确定性提议；后端执行链路是确定且可审计的工具调用。

### 7.2 前端模块划分

建议在前端引入一套 Explore 专属模块，并尽量复用现有代码：

| 模块                         | 复用 / 新增 | 职责                                                                           |
| ---------------------------- | ----------- | ------------------------------------------------------------------------------ |
| **AI Infra Adapter**         | 复用现有    | 基于 `useConfigStore` 的 AI 配置和 `OpenAIChat`，提供 Explore 专用模型调用接口 |
| **Explore Store**            | 新增        | 保存会话状态、工作记忆、当前提议、时间线、UI 状态、恢复快照                    |
| **Session Controller**       | 新增        | 驱动状态机和 Agent Loop，协调 Observe / Plan / Review / Execute / Verify       |
| **Observation Service**      | 新增        | 基于 `mfwProtocol` 获取截图、OCR、控制器状态、必要的候选区域数据               |
| **Prompt / Context Builder** | 新增        | 组装目标、历史、差异、风险、模板提示和输出契约                                 |
| **Proposal Normalizer**      | 新增        | 对模型输出做 JSON 解析、Schema 校验、字段修正与节点映射                        |
| **Execution Broker**         | 新增        | 统一封装直接 mfw 操作和基于 `DebugProtocol` 的临时调试执行                     |
| **Verification Service**     | 新增        | 根据验证计划做规则判断、轻量模型复核和人工确认分流                             |
| **Explore Modal UI**         | 新增        | 提供 FAB、协作 Modal、时间线、截图联动、Ghost Node 审阅和底部操作栏            |
| **Flow Bridge**              | 新增        | 把确认后的 `node_draft` 落地到现有 Flow Store 和历史系统                       |
| **Snapshot Manager**         | 新增        | 保存最近会话快照、关键截图引用和恢复入口                                       |

推荐最小新增工程形态：

- `Explore Store` 作为前端唯一事实源
- `ExploreAgentClient` 作为对 `OpenAIChat` 的 Explore 场景封装
- `ExploreModal` 作为主要 UI 容器

### 7.3 后端职责边界

LocalBridge 后端保持轻量、确定性的职责边界：

| 后端能力                       | 是否负责 | 说明                                               |
| ------------------------------ | -------- | -------------------------------------------------- |
| 控制器连接 / 状态维护          | **是**   | 继续使用现有控制器创建与连接状态能力               |
| 截图 / OCR / 原子动作          | **是**   | 继续通过 mfw 提供截图、OCR、点击、滑动、输入等能力 |
| 调试执行 / `pipeline_override` | **是**   | 继续复用现有 Debug 链路执行临时节点或临时链路      |
| 模型调用                       | **否**   | 不在后端维护 API Key、Prompt、会话或模型路由       |
| Agent 工作记忆                 | **否**   | 工作记忆和反馈吸收完全在前端                       |
| 提议生成 / 重规划              | **否**   | 所有提议与策略判断都在前端完成                     |
| Explore 专属会话服务           | **否**   | MVP 不新增后端 ExploreService，避免双端状态源冲突  |

### 7.4 通信与协议设计

MVP 建议使用“两条通信通道”：

1. **前端直连模型服务**：复用当前 AI 配置和 `OpenAIChat`
2. **前端通过 WebSocket 调用 LocalBridge**：复用现有 mfw / debug 协议

优先复用的现有通道建议如下：

| 通道 / 路由                                                           | 方向            | 用途                                     |
| --------------------------------------------------------------------- | --------------- | ---------------------------------------- |
| OpenAI-Compatible HTTP API                                            | 前端 → 模型服务 | 生成结构化提议、轻量复核、差异解释       |
| `/etl/mfw/request_screencap` → `/lte/mfw/screencap_result`            | 前端 ↔ 后端     | 获取当前截图                             |
| `/etl/utility/ocr_recognize` → `/lte/utility/ocr_result`              | 前端 ↔ 后端     | OCR 识别与 ROI 验证                      |
| `/etl/mfw/controller_click` / `swipe` / `input_text` / `start_app` 等 | 前端 → 后端     | 执行原子动作                             |
| `/mpe/debug/start` + `/lte/debug/*`                                   | 前端 ↔ 后端     | 执行临时节点 / 临时链路测试与调试回流    |
| `localStorage` / `IndexedDB`                                          | 前端本地        | 保存会话快照、恢复入口、关键时间线元数据 |

MVP 阶段不建议新增 `/mpe/explore/*` 这类后端专属 Explore 会话协议；如果后续确实需要团队共享、跨设备恢复或远端协作，再考虑单独引入。

### 7.6 Ghost Node 的工程落地策略

建议采用以下策略：

- **不新增持久化节点类型**，保持正式节点仍为现有 Pipeline Node
- 前端单独维护 `draftMeta`，如 `isGhost`、`proposalId`、`confidence`、`riskLevel`、`overlayBindings`
- 只有在用户确认后，才把 `node_draft` 写入现有 Flow Store
- 未确认的 Ghost Node 不参与正常保存与导出

这样既能复用现有节点渲染与字段编辑能力，也能降低对当前画布模型的侵入性。

### 7.7 持久化、日志与回放

建议从一开始就为可恢复性预留前端快照结构：

- **会话快照**：保存最近一次会话状态、目标、步骤号、工作记忆和已确认链
- **截图缓存**：保留最近若干步的截图引用、标注信息和验证结果
- **审阅日志**：记录用户对每个提议的确认、修改、拒绝、回退和接管
- **Agent Trace**：记录 Observe / Plan / Review / Execute / Verify / Replan 的关键事件
- **回放能力**：后续可基于日志回放探索过程，用于调试、产品改进和模板沉淀

MVP 可以先做“最近会话恢复”，完整回放可放后续阶段。存储建议优先本地化：轻量元数据放 `localStorage`，较大快照或图片引用再考虑 `IndexedDB`。

## 8. 与现有功能的结合

### 8.1 与现有前端 AI Infra 的结合

- 复用 `useConfigStore` 中的 `aiApiUrl / aiApiKey / aiModel` 配置项
- 复用 `src/utils/openai.ts` 中的 `OpenAIChat` 会话封装能力，避免再造一层通用 AI SDK
- 复用 `aiHistoryManager` 记录探索模式下的重要模型交互，便于调试和问题定位
- 配置入口继续复用现有 `AIConfigSection`，避免新增一套 Explore 专属 AI 设置页
- MVP 官方推荐用户配置支持 CORS 的 OpenAI-Compatible 中转服务，继续沿用现有本地风险提示

### 8.2 LocalBridge 与 mfw 能力复用

- 复用现有截图、OCR、点击、滑动、输入、启动应用、停止应用等能力
- RLHF AI 辅助流程探索模式不重新发明执行层，只在前端新增会话、工作记忆和决策编排层
- 执行结果继续沿用既有协议风格，降低集成成本

### 8.3 与现有调试能力的结合

- “先测试”可复用现有 `DebugProtocol` 和 `pipeline_override`，把当前 `node_draft` 临时拼成最小执行链路
- 对需要完整节点语义的验证，优先走调试执行而不是前端自己模拟
- 调试事件流可以映射进探索时间线，作为“工具消息”或“执行消息”展示

### 8.4 字段面板与 Modal 工作台的结合

- 当前 Ghost Node 可直接复用字段编辑器进行参数微调
- 进入探索模式后，字段编辑器应以内嵌 Tabs 的形式挂载到协作 Modal 右侧草稿区，而不是继续依赖右侧固定侧栏
- 用户修改后的字段应回写到当前提议，并覆盖 Agent 原始输出
- 现有智能补全能力继续作为字段级辅助存在，但从属于流程级智能体而不是并列入口

### 8.5 模板系统与维护场景结合

- Agent 提议可优先参考已有模板和历史节点模式
- 用户确认且验证通过的高质量步骤可沉淀为模板候选
- 在维护场景中（Phase 2），可展示"已有节点摘要 + 新提议差异摘要"，完整并排 `差异对比` 页签放到 M4

### 8.6 现有 Store 与画布逻辑结合

建议新增独立的 `Explore Store`，而不是把探索状态硬塞进现有 Flow Store：

- **Explore Store**：会话状态、Agent 状态、提议、时间线、截图、标注、Modal UI、恢复信息
- **Flow Store**：正式节点、边、历史、视图、路径等既有编辑状态

两者的交界点只有一个：

> 当用户确认当前提议时，才把 `node_draft` 写入 Flow Store，并纳入正常历史记录。

## 9. 验收标准与指标

### 9.1 功能验收

MVP 至少应满足以下能力闭环：

1. 已连接控制器后，可以通过右下角 AI 入口成功打开 RLHF 协作 Modal，并创建探索会话或恢复最近会话
2. 协作 Modal 可以持续展示截图区、对话时间线、当前提议、草稿字段、差异信息和底部操作区
3. 前端可以基于当前 AI 配置直接完成模型调用，并生成结构化提议、截图标注和验证计划
4. 用户可以确认、修改、拒绝、回退、接管、终止当前步骤，并可通过自然语言补充修正
5. 确认后可以通过现有 mfw / debug 能力执行该步骤，并获得执行结果、验证结果与新截图
6. 验证失败时，系统能进入重规划或要求人工介入，而不是继续盲跑
7. 失败、断连、截图异常时，系统能给出清晰反馈并支持恢复或终止
8. 页面刷新后，若存在前端快照，可以恢复最近会话的目标、步骤、工作记忆和关键时间线

### 9.2 体验指标

建议关注以下指标，用于衡量该模式是否真正有价值：

- **首个可运行流程完成时间**
- **单步提议确认率**
- **单步人工修改率**
- **验证失败后重规划成功率**
- **因执行失败触发回退的比例**
- **从截图到提议返回的耗时**
- **会话中断后的恢复成功率**

### 9.3 实施里程碑建议

| 里程碑 | 内容                                                              |
| ------ | ----------------------------------------------------------------- |
| **M1** | 定义 Explore Store、前端 Agent Runtime、数据结构和协作 Modal 骨架 |
| **M2** | 打通观测、上下文拼装、单步提议展示、Ghost Node 和审阅交互         |
| **M3** | 打通测试 / 执行 / 验证 / 重规划闭环，并接入现有 debug 能力        |
| **M4** | 增强恢复能力、维护适配、模板沉淀与差异视图                        |

## 10. 价值与愿景

### 10.1 用户价值

| 用户类型       | 获得价值                                |
| -------------- | --------------------------------------- |
| **新手**       | 降低学习门槛，在引导下完成首个 Pipeline |
| **熟练开发者** | 提升编写效率，减少机械重复劳动          |
| **维护者**     | 更快定位失效步骤，降低 UI 适配成本      |

### 10.2 长期愿景

- **把 AI 从补全器提升为协作 Agent**：它不只是帮用户填字段，而是帮用户推进任务、验证结果、处理偏差
- **个人知识沉淀**：用户的确认、修改与拒绝记录可形成个人偏好素材
- **团队知识共享**：高质量探索结果可沉淀为团队模板与最佳实践
- **生态共建**：未来可形成基于场景的探索策略库，而不是只沉淀零散节点

## 11. 相关概念区分

| 概念                | 说明                                           | 与本模式的关系                                                                 |
| ------------------- | ---------------------------------------------- | ------------------------------------------------------------------------------ |
| **提示词工程**      | 主要关注单次模型调用的输入组织与输出格式       | 本模式会使用 prompt，但 prompt 只是 Agent Runtime 的一个实现组件，不是整体架构 |
| **传统 AI 补全**    | 单节点级别的字段补全                           | 本模式是流程级协作，传统补全仍作为子能力存在                                   |
| **单 Agent 类架构** | 一个有状态、可调用工具、可吸收反馈的受约束代理 | 本模式当前阶段的推荐实现方式                                                   |
| **多 Agent 编排**   | 多个代理彼此分工协作                           | 当前阶段不是必需，待单 Agent 跑稳后再评估                                      |
| **MCP**             | 模型上下文协议，强调标准化工具调用             | 本模式当前不依赖 MCP，优先复用 LocalBridge 直接调用 mfw 的能力                 |
| **Skills / 模板**   | 文档、脚本或结构化经验的封装                   | 探索结果未来可进一步沉淀为模板、策略或 Skill                                   |
| **普通编辑模式**    | 用户直接在画布上手动编排节点                   | 探索模式是增强层，可随时切换回普通编辑模式                                     |

## 12. 关键决策定稿

以下问题在本规划中统一定稿，不再作为开放项：

| 决策项                     | 最终方案                                                                                    | 说明                                                                                                                                                  |
| -------------------------- | ------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **MVP 默认模型路线**       | 默认接入 **OpenAI-Compatible** 的云端聊天模型，不优先本地模型                               | 首要目标是低延迟、结构化输出稳定、可直接复用现有前端 AI infra；本地模型作为可选实验路线保留，但不作为首版默认路径                                     |
| **Explore 专用 AI Client** | **是**，在 `OpenAIChat` 之上封装一层 `ExploreAgentClient`                                   | 只做薄封装，负责系统提示词模板、结构化 JSON 输出约束、取消请求、限流、重试和探索态历史裁剪；不重写底层 AI infra                                       |
| **工作记忆与快照存储**     | 运行态只放前端内存；轻量快照放 `localStorage`；大对象放 `IndexedDB`                         | `Explore Store` 作为运行时单一事实源；目标、步骤、工作记忆摘要和时间线元数据进入 `localStorage`；截图引用和较大缓存进入 `IndexedDB`                   |
| **API Key / CORS 策略**    | MVP 继续采用前端直连模型服务，不新增后端代理                                                | 官方推荐用户配置支持 CORS 的 OpenAI-Compatible 中转服务；同时保留现有“API Key 明文存本地”的风险提示，不在 MVP 阶段把密钥托管到 LocalBridge            |
| **模型前预处理层**         | **要做轻量预处理**，不做重型多阶段视觉流水线                                                | 每轮模型输入固定包含：当前截图摘要、OCR 结果、候选框 / 关键区域、页面状态摘要；维护场景可选补充已有节点差异信息；不引入复杂 CV 编排或额外后端感知服务 |
| **“先测试”默认路径**       | 优先级固定为：**纯识别验证 → 低风险 direct mfw 原子动作 → `pipeline_override` 调试验证**    | 先用无副作用方式验证；只有在需要真实动作或完整节点语义时才升级到 direct mfw 或 debug 执行，避免测试默认产生不必要副作用                               |
| **模板沉淀时机**           | `确认 + 验证通过 + 非敏感` 后进入“模板候选”；满足“重复成功 2 次”或用户手动 Pin 后才正式入库 | 先收集候选，再做正式沉淀，降低噪音模板比例；敏感步骤默认不自动沉淀                                                                                    |
