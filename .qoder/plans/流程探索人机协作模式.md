# MPE 流程探索人机协作模式

## 1. 背景与动机

### 1.1 当前痛点

在 MaaFramework Pipeline 的开发过程中，开发者通常会遇到以下问题：

- **古法编写效率低**：需要手动截图、逐个配置节点、反复调试验证
- **AI 辅助视野有限**：现有 AI 补全大多停留在单节点或字段级别，缺少“流程下一步该做什么”的整体判断
- **验证周期长**：往往要把一串步骤写完后再整体运行，问题发现得晚、回溯成本高
- **维护成本高**：一旦游戏 UI 改版，旧流程容易局部失效，但排查通常只能靠人工逐步试错

### 1.2 核心洞察

借鉴 RLHF（Reinforcement Learning from Human Feedback）中的“人在回路”思想，但不追求自动训练本身，而是强调协作闭环。其核心为，问题的难点不在于“让模型生成一个节点”，而在于：

- 当前画面是动态的、不完全可观测的
- 下一步动作依赖前序步骤、执行结果和用户修正
- 执行后必须校验结果，失败后要能够重规划而不是继续硬跑
- 涉及敏感动作、歧义目标、不可逆操作时必须让人保留控制权

因此，这个能力不应被定义为“提示词工程增强版”，而应被定义为一个**有状态、可调用工具、可被人审阅和打断的 Explore Agent Runtime**。

> 真正需要设计的不是一条 prompt，而是一套“观测 → 记忆 → 规划 → 提议 → 审阅 → 执行 → 验证 → 重规划”的协作运行机制。

### 1.3 设计原则

- **先提议，后落地**：Agent 产出的是待确认提议，不直接写入正式流程
- **单 Agent 优先，角色分层实现**：MVP 先做一个会话型 Explore Agent，在内部拆分感知、记忆、规划、验证等职责，而不是一开始就上多 Agent 编排
- **每步可审阅、可中断、可回退**：任何一步都应可观察、可修改、可撤销、可恢复
- **结构化优先**：Agent 输出应尽可能是稳定的数据结构，而非自由文本
- **人始终拥有最终决定权**：涉及歧义、高风险、不可逆动作时必须保留人工把关
- **Prompt 只是实现细节**：真实能力边界来自 prompt、工具契约、状态机、工作记忆、安全规则和审阅闭环的组合
- **与现有工作流共存**：探索模式增强编辑体验，不替代现有的字段编辑、调试、模板与手工编排能力

## 2. 模式定义

### 2.1 什么是流程探索人机协作模式

这是一种**流程级的人机协作工作流**。其中的 AI 核心，不再被视为一次性的“提词器”，而是一个**会话范围内受约束的 Explore Agent**。

| 角色                  | 职责                                                                                             |
| --------------------- | ------------------------------------------------------------------------------------------------ |
| **用户**              | 给出目标、审阅提议、修正歧义、决定是否执行、在必要时手动接管                                     |
| **Explore Agent**     | 根据当前画面、历史上下文、已确认步骤和用户反馈，决定“下一步该做什么”，并输出结构化提议与备选方案 |
| **MPE**               | 提供可视化交互环境，承载探索会话、截图标注、Ghost Node 预览、提议审阅和流程落地                  |
| **LocalBridge / mfw** | 提供截图、识别、点击、滑动、输入、等待等底层能力，并将确认后的步骤真正执行到设备上               |

### 2.2 为什么这是类 Agent 架构，而不是提示词工程

如果把这个问题当成“截图 + prompt + JSON 输出”，会天然漏掉以下核心能力：

- **会话状态**：同一个目标要跨越多轮截图、多个步骤和多次人工修正
- **工具使用**：模型输出不是终点，必须联动截图、识别、点击、测试、执行等工具
- **短程规划**：需要在当前上下文下决定主路径、备选路径、验证条件和终止条件
- **反馈吸收**：用户修改 ROI、否决提议、手动插入步骤后，后续决策必须受影响
- **失败恢复**：执行结果不符合预期时，需要进入重规划而不是继续沿原路径前进
- **安全守卫**：敏感动作、高风险动作、不可逆动作必须受到规则限制

因此，这里的“agent”不是指一个开放式、无限自治的系统，而是指一个**受约束、短视距、强审阅的人机协作代理**。

### 2.3 Explore Agent 的内部组成

推荐将 Explore Agent 设计为一个单运行时、内部多职责模块的结构：

| 模块                         | 职责                                                                     |
| ---------------------------- | ------------------------------------------------------------------------ |
| **Perception / Observation** | 获取截图、OCR、目标候选区域、控件上下文等观测信息                        |
| **Working Memory**           | 保存目标、当前步骤、已确认链路、用户修正、失败记录、安全标记等会话内记忆 |
| **Planner / Policy**         | 根据当前观测和工作记忆生成短程计划，决定当前主提议与有限备选             |
| **Proposal Assembler**       | 将决策结果转换为前端可审阅的结构化提议、Ghost Node 和截图标注            |
| **Review Adapter**           | 吸收用户的确认、修改、拒绝、回退、手动接管等反馈，回写工作记忆           |
| **Execution Supervisor**     | 区分测试与正式执行，调用 mfw 原子能力并收集执行回流                      |
| **Verifier / Critic**        | 判断执行结果是否符合预期，失败时触发重规划或人工介入                     |
| **Safety Guard**             | 检查敏感动作、不可逆动作、低置信度、高歧义等情况，决定是否必须人工确认   |
| **Trace Recorder**           | 记录观测、提议、审阅、执行、验证和恢复信息，用于回放、调试和模板沉淀     |

### 2.4 核心交互闭环

```text
用户设定目标
      ↓
Observe：LocalBridge 获取当前画面与上下文
      ↓
Update Memory：写入工作记忆（目标、历史、用户修正、失败信息）
      ↓
Plan：Explore Agent 进行短程规划，产出主提议和有限备选
      ↓
Propose：MPE 呈现 Ghost Node + 截图标注 + 决策摘要
      ↓
Review：人审阅并操作 [确认] [修改] [拒绝] [测试] [回退] [接管] [终止]
      ↓
Execute / Test：通过 LocalBridge / mfw 执行或验证当前步骤
      ↓
Verify：判断结果是否符合预期
      ↓
成功 → 写入 confirmed_chain → 下一轮探索
失败 → 重规划 / 人工介入 / 终止
```

### 2.5 范围边界

本模式：

- 不是“全自动无人值守脚本生成器”
- 不是对普通编辑模式的替代，而是其增强层
- 不是单节点 AI 补全的简单放大版，而是带有会话、状态、工具调用和执行闭环的流程协作模式
- 不是一开始就做“多 Agent 自组织系统”，MVP 先做单 Agent Runtime，把工程可控性放在第一位

## 3. 目标用户与典型场景

### 3.1 场景一：新手快速上手

**用户**：刚接触 MaaFramework 的新开发者  
**目标**：完成游戏登录流程

**流程**：

1. 用户在 MPE 中输入目标：“完成游戏登录”
2. Explore Agent 分析当前游戏画面，提议第一步：“点击账号输入框”
3. MPE 在画布上显示半透明 Ghost Node，并在截图中高亮推荐区域
4. 用户确认位置正确，或微调 ROI 后点击“确认”
5. 节点变为正式节点，LocalBridge 执行该步骤
6. 执行结果回流后，Explore Agent 更新工作记忆，继续推进入下一步

**价值**：新手无需先理解 MaaFramework 的字段细节，也能在引导下完成首个可运行 Pipeline。

### 3.2 场景二：复杂流程拆解

**用户**：有经验的开发者  
**目标**：完成游戏内一个多步骤活动

**流程**：

1. 用户输入目标：“完成每日签到活动”
2. Explore Agent 识别出关键路径：点击活动入口 → 切换签到标签 → 点击签到按钮 → 关闭奖励弹窗
3. 每一步都在 MPE 中以可审阅提议呈现，用户可根据实际情况微调
4. 遇到分支时，例如“今日已签到”或“活动入口位置变化”，Agent 给出主提议和有限备选
5. 执行偏离预期时，系统进入验证失败和重规划，而不是静默继续

**价值**：把复杂流程中的“探索脑力”转移给 Agent，人主要负责异常判断与结果把关。

### 3.3 场景三：流程维护与适配

**用户**：需要维护已有 Pipeline 的开发者  
**目标**：适配游戏 UI 改版

**流程**：

1. 用户加载旧版 Pipeline，启动探索模式
2. Explore Agent 对比旧节点配置与当前画面，识别失效步骤或高风险步骤
3. 对失效步骤重新提议识别区域与动作参数，并给出与旧节点的差异摘要
4. 用户逐条审阅并确认新的适配方案

**价值**：降低 UI 改版后的维护成本，帮助用户更快定位失效点并完成修复。

### 3.4 场景四：人机接力编辑

**用户**：熟悉编辑器但不想被 AI 绑住流程的开发者  
**目标**：在关键步骤由 AI 提议，在细节步骤手工接管

**流程**：

1. 用户让 Explore Agent 先完成主路径探索
2. 在某个复杂节点处切换到普通编辑模式，手动调整参数
3. 保存后继续探索，Agent 将手动编辑结果视为新的权威上下文

**价值**：探索模式不会“锁死”用户，而是可以自然退回现有编辑工作流。

## 4. 产品目标与阶段范围

### 4.1 产品目标

- 降低从“有目标”到“拿到首个可运行流程”的门槛
- 缩短流程编写与验证之间的反馈回路
- 降低复杂流程和维护场景中的认知负担
- 把 AI 能力从字段补全提升到流程级协作 Agent

### 4.2 MVP 范围

MVP 聚焦最小可用闭环，核心不是“做一个很会说的 prompt”，而是“做一个可控的会话型 Agent Runtime”：

- 单控制器、单活跃探索会话
- 单个 Explore Agent Runtime
- 短视距规划：每轮产出 **1 个主提议 + 最多 2 个备选**
- 单步提议 → 人确认或修改 → 测试 / 执行 → 验证 → 下一步 / 重规划
- 支持基础动作类型：点击、滑动、输入、等待、判断、结束、请求人工介入
- 支持截图标注、Ghost Node 预览、决策摘要展示
- 支持回退一步、拒绝当前提议、手动接管、终止探索
- 支持最近会话恢复
- 支持结构化提议契约、工作记忆和基础安全规则

### 4.3 暂不纳入 MVP 的内容

- 多控制器并行探索
- 多 Agent 协作编排
- 长链路全自动无人值守生成
- 大规模并行分支树生成
- 对旧 Pipeline 的全量自动迁移
- 基于用户历史自动训练个人偏好模型

### 4.4 阶段演进建议

| 阶段                | 目标                  | 核心能力                                                       |
| ------------------- | --------------------- | -------------------------------------------------------------- |
| **Phase 1**         | 打通单 Agent 探索闭环 | 会话管理、工作记忆、单步提议、Ghost Node、确认执行、验证与回退 |
| **Phase 2**         | 提升复杂场景可用性    | 分支候选、失败重规划、维护适配、旧节点对比、模板辅助           |
| **Phase 3**         | 形成知识沉淀能力      | 会话回放、模板沉淀、偏好记忆、团队共享                         |
| **Phase 4（可选）** | 评估更强 agentic 能力 | Planner/Critic 拆分、多阶段策略、有限多 Agent 协作             |

说明：

- **多 Agent 不是 MVP 的必要条件**
- 先把单 Agent Runtime 做稳，再决定是否拆出 Planner / Critic 等子代理

## 5. 交互与状态设计

### 5.1 探索会话模型

探索模式应被建模为一个独立的 **Explore Session**，其核心不是“当前 prompt 内容”，而是“当前 agent 在这个会话里的状态”。

| 状态项                  | 说明                                                                                   |
| ----------------------- | -------------------------------------------------------------------------------------- |
| **session_id**          | 当前探索会话唯一标识                                                                   |
| **controller_id**       | 当前关联的设备或控制器                                                                 |
| **goal**                | 当前探索目标描述                                                                       |
| **status**              | 会话生命周期状态，例如 active / paused / completed / failed                            |
| **agent_state**         | Agent 当前内部阶段，如 observing / planning / awaiting_review / verifying / replanning |
| **current_step**        | 当前正在处理的第 N 步                                                                  |
| **working_memory**      | 会话内工作记忆，包含已确认链、用户修正、失败信息、约束条件                             |
| **current_plan**        | 当前短程计划，包括主提议、备选、预期结果和验证条件                                     |
| **proposal**            | 当前待审阅的结构化提议                                                                 |
| **screenshot_context**  | 当前步骤关联的截图、ROI、标注信息                                                      |
| **confirmed_chain**     | 已确认并落地的步骤序列                                                                 |
| **review_history**      | 用户确认、修改、拒绝、回退、接管等操作记录                                             |
| **verification_result** | 最近一次执行 / 测试后的验证结果                                                        |
| **safety_flags**        | 当前是否存在敏感、高风险、不可逆等安全标记                                             |
| **last_error**          | 最近一次失败信息或警告                                                                 |

额外约束：

- 同一时刻一个控制器只允许存在一个活跃探索会话
- 会话状态应独立于普通画布编辑状态，避免污染既有文件编辑逻辑
- 用户手动修改后的结果必须高优先级写入 `working_memory`

### 5.2 探索状态机

需要显式定义探索模式的状态迁移，避免“UI 看起来能点，但后台语义不清”的情况：

| 状态                   | 含义                             | 允许迁移                                               |
| ---------------------- | -------------------------------- | ------------------------------------------------------ |
| **idle**               | 尚未开始探索                     | observing / terminated                                 |
| **observing**          | 正在获取截图与上下文             | planning / failed                                      |
| **planning**           | 正在基于工作记忆生成短程计划     | reviewing_proposal / failed                            |
| **reviewing_proposal** | 用户正在审阅当前提议             | testing / executing / replanning / paused / terminated |
| **testing**            | 用户选择先测试当前步骤           | verifying / failed                                     |
| **executing**          | 已确认并正在正式执行步骤         | verifying / failed                                     |
| **verifying**          | 正在判断结果是否符合预期         | observing / replanning / completed / failed            |
| **replanning**         | 因失败、拒绝或环境变化而重新规划 | reviewing_proposal / paused / failed                   |
| **paused**             | 用户手动接管或暂时中止           | observing / planning / terminated                      |
| **failed**             | 当前步骤或会话失败               | replanning / paused / terminated                       |
| **completed**          | 目标达成，探索结束               | idle                                                   |
| **terminated**         | 用户主动终止                     | idle                                                   |

补充说明：

- 页面刷新或 WebSocket 重连后，若检测到未结束会话，可提供“恢复最近会话”入口
- 一旦控制器断连、截图失败、执行层不可用，会话应自动进入 `failed` 或 `paused`
- `replanning` 是 agent 架构的重要状态，不能被简化成再次请求同一个 prompt

### 5.3 Working Memory 建模

工作记忆是 Explore Agent 区别于“单轮提示词”的关键。建议至少拆分为以下几类：

| 记忆类型                   | 内容                                             |
| -------------------------- | ------------------------------------------------ |
| **Goal Memory**            | 用户目标、范围限制、完成条件                     |
| **Interaction Memory**     | 已确认步骤、当前已进入的页面、关键上下文变量     |
| **User Correction Memory** | 用户手动调整的 ROI、参数、命名、分支选择偏好     |
| **Failure Memory**         | 最近失败的动作、失败原因、禁止重复尝试的错误路径 |
| **Safety Memory**          | 敏感页面、不可逆动作、必须二次确认的情形         |
| **Template Hint Memory**   | 来自模板、历史节点、现有 Pipeline 的偏置提示     |

记忆写入规则建议：

- 用户明确修改过的内容优先级高于模型原始判断
- 执行失败但未确认原因的信息只作为短期记忆，不应直接沉淀为模板
- 只有“确认 + 验证通过”的高质量步骤，才有资格进入长期沉淀

### 5.4 Ghost Node（幽灵节点）

Ghost Node 是 Agent 当前提议的可视化承载物，推荐采用**“现有 Pipeline Node + Draft 元信息”**的方式落地，而不是直接新增一种可持久化节点类型。

原因：

- 现有画布节点体系已经稳定，正式节点的数据结构也已明确
- Ghost Node 本质上是“待确认渲染态”，不是最终存档态
- 未确认提议不应进入正式 Pipeline JSON，也不应污染正常的导出与保存逻辑

推荐表现：

- 半透明或降低饱和度
- 虚线边框，显式区分“待确认”
- 显示置信度、动作类型、关键识别信息、风险级别
- 支持拖拽、框选调整、双击编辑字段
- 提供显式操作：确认、修改、拒绝、测试

### 5.5 截图联动标注

Agent 的提议必须与截图强绑定，至少包含：

- 当前 `screenshot_id`
- Agent 推荐的 ROI / target / begin / end 区域
- 候选目标框、置信度和标签
- 用户人工修正后的最终区域
- 预期执行结果的验证区域或验证条件

交互要求：

- 用户可拖动调整框选范围
- 用户可重新框选并把结果回灌给 Agent，或直接覆盖当前提议
- 一旦用户手动修改 ROI，该修改应视为当前步骤的权威输入

### 5.6 探索控制面板

建议在界面一侧提供固定控制面板，包含：

- **目标显示**：当前任务目标和控制器信息
- **Agent 状态显示**：当前处于 observing / planning / reviewing / verifying 等哪个阶段
- **进度显示**：第 N 步 / 预计总步数
- **决策依据摘要**：展示简短的选择理由、风险级别和置信度，不展示冗长内部推理
- **操作区**：确认、修改、拒绝、回退、测试、接管、终止
- **历史缩略图**：最近若干步截图与结果回顾
- **状态提示**：当前会话状态、错误信息、风险提醒、恢复入口

### 5.7 回退、接管与恢复

探索模式不仅要支持“向前走”，也必须明确“如何安全撤回”：

- **确认**：把 Ghost Node 正式写入当前流程，并记录到会话历史和工作记忆
- **回退一步**：撤销最近一次确认，同时通知后端回滚会话上下文
- **拒绝当前提议**：保留当前步骤号，但要求 Agent 基于同一画面和拒绝原因重新给出新方案
- **手动接管**：临时切回普通编辑逻辑，用户修改后再继续探索
- **恢复会话**：断连或刷新后，允许恢复最近会话的目标、步骤、工作记忆和历史截图

### 5.8 分支处理

MVP 阶段建议严格控制复杂度：

- 默认只展示 **1 个主提议 + 最多 2 个备选提议**
- 备选提议需要明确其触发前提，而不是只列出不同动作
- 用户先在备选中做选择，再继续后续探索
- “一次生成完整分支树”放到后续阶段，而不是 MVP 强行覆盖

## 6. Explore Agent 决策与执行模型

### 6.1 Agent Loop

建议将核心运行过程显式建模为以下循环：

| 阶段                  | 输入                           | 输出                                   |
| --------------------- | ------------------------------ | -------------------------------------- |
| **Observe**           | 当前截图、设备状态、已确认链路 | 标准化观测结果                         |
| **Update Belief**     | 观测结果 + 工作记忆            | 当前页面假设、关键变量、风险标记       |
| **Plan**              | 当前 belief + 用户目标         | 主提议、有限备选、预期结果、验证计划   |
| **Propose**           | 计划结果                       | 结构化提议、Ghost Node、截图标注、摘要 |
| **Review Align**      | 用户确认 / 修改 / 拒绝 / 接管  | 更新后的提议或重规划请求               |
| **Execute / Test**    | 已确认提议                     | 执行回流、前后截图、异常信息           |
| **Verify**            | 执行回流 + 预期结果            | 成功 / 失败 / 不确定                   |
| **Replan / Complete** | 验证结论                       | 下一轮探索、人工介入或结束             |

### 6.2 规划策略

Explore Agent 的规划建议遵循以下原则：

- **短视距规划**：默认只规划当前一步及有限备选，不追求一次性生成整条长链路
- **显式预期结果**：每个提议除了“做什么”，还要包含“做完后应该看到什么”
- **失败可重规划**：如果验证结果不匹配，应优先重规划，而不是继续沿旧路径推进
- **人工优先兜底**：不确定时，优先输出“请求人工确认/接管”，而不是硬猜

Agent 的动作意图建议统一收敛到有限类别：

- `act`
- `wait`
- `assert`
- `branch`
- `finish`
- `ask_human`

### 6.3 Agent 输出约束

为保证稳定性，Agent 输出应尽量是**结构化 JSON**，而不是自由文本。至少应包含：

- `proposal_id`
- `step_index`
- `proposal_kind`
- `action_type`
- `recognition_type`
- `node_draft`
- `overlays`
- `confidence`
- `risk_level`
- `reasoning_summary`
- `expected_outcome`
- `verification_plan`
- `alternatives`
- `needs_human_attention`

其中：

- `proposal_kind` 用于区分 `act / wait / assert / branch / finish / ask_human`
- `node_draft` 应尽量贴合现有 Pipeline 节点结构
- `expected_outcome` 用于描述执行后应观察到的页面变化或识别条件
- `verification_plan` 用于指导执行后如何判定成功
- `reasoning_summary` 只保留对用户有帮助的简短摘要，不暴露冗长内部推理
- `alternatives` 仅保留有限个候选方案，避免 UI 过载

### 6.4 用户反馈如何写回 Agent

用户反馈不应只影响当前 UI，而应回写到 Agent 的工作记忆和策略上下文：

- **确认**：将当前提议写入 `confirmed_chain`，并标记为已验证候选
- **修改**：以用户修改结果覆盖当前提议，并写入 `User Correction Memory`
- **拒绝**：记录拒绝原因，避免 Agent 在同一画面重复给出相同低质量方案
- **手动接管**：把用户手动编辑的节点视为当前步骤的权威结果
- **回退**：同步回滚会话上下文和最近一次记忆写入

### 6.5 置信度与人工介入规则

建议按置信度分层：

- **高置信度**：可以高亮推荐“确认”，但仍保留人工最终确认
- **中置信度**：默认引导用户检查 ROI、关键参数和验证条件
- **低置信度**：默认推荐人工重框选、修改或切换手动编辑

以下场景应直接要求人工介入，而不是继续自动推进：

- 屏幕元素识别冲突严重
- 动作目标不唯一
- 执行后画面变化与预期不一致
- 当前步骤涉及账号、密码、支付、删除、退出登录等敏感动作
- 当前 Agent 无法给出明确的预期结果和验证方式

### 6.6 安全边界

为了避免“AI 看起来聪明，但实际越界”：

- 涉及**不可逆动作**时必须显式二次确认
- 涉及**敏感输入**时应优先手动接管，或在用户明确授权后才执行
- 当 Agent 无法解释为何选择某一步时，不应允许其默默执行
- 当设备状态异常、控制器断连、截图失败、执行层报错时，会话必须中断并提示用户

### 6.7 测试、执行与验证策略

建议将“测试”“正式执行”“结果验证”拆成三个独立阶段：

- **先测试**：只验证当前提议在当前画面是否成立，适合高风险或低置信度步骤
- **正式执行**：在用户确认后，真正对设备发送动作
- **结果验证**：根据 `expected_outcome` 和 `verification_plan` 判断是否成功

验证失败后不应直接报错结束，而应优先：

1. 写入 `Failure Memory`
2. 进入 `replanning`
3. 给出新的主提议或请求人工介入

### 6.8 学习与沉淀策略

MVP 不做在线训练，但应为后续沉淀预留结构：

- 记录用户确认、修改、拒绝、回退的数据
- 记录哪些提议最终通过验证，哪些提议最终失败
- 将高质量的“确认 + 验证通过”步骤沉淀为模板候选
- 后续可基于这些轨迹构建模板偏置、偏好记忆和维护修复建议

## 7. 技术架构设计

### 7.1 总体原则

**决策**：Explore Agent Runtime 由 LocalBridge 后端直接承载，复用现有 mfw 能力和会话型服务模式，不引入额外 MCP 层。

原因：

- 探索会话本质上是“截图 → Agent 提议 → 人确认 → 执行 → 验证 → 重规划”的短闭环
- LocalBridge 已具备成熟的 mfw 调用链路，截图、点击、识别等能力可直接复用
- 现有 `DebugSession` 风格已经证明“后端维护会话、前端订阅事件”的模式是可行的
- 后端更适合维护多轮上下文、缓存截图、管理模型调用和安全策略
- 前端无需承担截图中转、模型调用和敏感状态存储职责

更准确的表述应是：

> 执行链路是确定且可审计的，Agent 决策是受约束的非确定性提议。

### 7.2 后端模块划分

推荐在 LocalBridge 内新增一套与 `DebugServiceV2` 平行的 Explore 服务：

| 模块                      | 职责                                                          |
| ------------------------- | ------------------------------------------------------------- |
| **ExploreService**        | 对外提供会话创建、查询、暂停、恢复、终止等能力                |
| **ExploreSessionManager** | 管理会话生命周期、状态迁移和单控制器唯一活跃约束              |
| **ExploreAgentRuntime**   | 驱动 Observe / Plan / Review / Execute / Verify / Replan 循环 |
| **PerceptionAdapter**     | 封装截图、OCR、目标候选提取、页面上下文分析                   |
| **WorkingMemoryStore**    | 管理会话内工作记忆、用户修正、失败记录和安全标记              |
| **PlannerPolicy**         | 组织模型输入、约束模型输出、生成短程计划                      |
| **ProposalAssembler**     | 生成前端可渲染的提议结构、截图标注和 Ghost 元信息             |
| **SafetyGuard**           | 校验敏感动作、低置信度、高歧义、不可逆动作等风险              |
| **ExecutionOrchestrator** | 统一调用测试与正式执行，收集执行回流                          |
| **VerificationEngine**    | 根据预期结果和验证计划判断是否成功                            |
| **TraceRecorder**         | 保存事件流、截图、审阅日志和恢复快照                          |

### 7.3 前端职责划分

| 层级           | 模块                | 职责                                                     |
| -------------- | ------------------- | -------------------------------------------------------- |
| **前端 (MPE)** | 探索模式 UI         | 探索面板、Agent 状态展示、Ghost Node、截图标注交互       |
|                | Explore Store       | 维护会话状态、当前提议、审阅操作、恢复信息和最近验证结果 |
|                | Flow 集成层         | 把确认后的步骤落地到现有画布和历史系统                   |
|                | Review Controller   | 将用户确认、修改、拒绝、回退、接管等操作转换为协议消息   |
|                | Compare / Diff 视图 | 在维护场景中展示旧节点与新提议的差异                     |

### 7.4 探索协议设计

建议沿用现有 `path + data` 的 WebSocket 通信风格，并和当前 `/mpe/debug/*` / `/lte/debug/*` 风格保持一致，为探索模式定义独立协议：

| 路由                                | 方向        | 用途                                       |
| ----------------------------------- | ----------- | ------------------------------------------ |
| `/mpe/explore/create_session`       | 前端 → 后端 | 创建探索会话并提交目标                     |
| `/lte/explore/session_created`      | 后端 → 前端 | 返回 `session_id` 与初始状态               |
| `/mpe/explore/request_next`         | 前端 → 后端 | 请求基于当前画面生成下一步提议             |
| `/lte/explore/observation`          | 后端 → 前端 | 返回当前观测摘要、截图 ID 和上下文信息     |
| `/lte/explore/proposal`             | 后端 → 前端 | 返回当前步骤提议、标注、候选方案和摘要     |
| `/mpe/explore/review`               | 前端 → 后端 | 提交确认、修改、拒绝、回退、接管等审阅结果 |
| `/mpe/explore/test_step`            | 前端 → 后端 | 请求先测试当前提议                         |
| `/mpe/explore/execute_step`         | 前端 → 后端 | 请求正式执行当前提议                       |
| `/lte/explore/execution_result`     | 后端 → 前端 | 返回测试或正式执行结果                     |
| `/lte/explore/verification_result`  | 后端 → 前端 | 返回验证结论、偏差说明和是否需要重规划     |
| `/mpe/explore/pause`                | 前端 → 后端 | 暂停当前会话                               |
| `/mpe/explore/resume`               | 前端 → 后端 | 恢复当前会话                               |
| `/mpe/explore/restore_last_session` | 前端 → 后端 | 恢复最近未完成会话                         |
| `/mpe/explore/terminate`            | 前端 → 后端 | 终止当前会话                               |
| `/lte/explore/state_changed`        | 后端 → 前端 | 推送会话状态变化                           |
| `/lte/explore/error`                | 后端 → 前端 | 返回可恢复或不可恢复错误                   |

核心字段建议统一包含：

- `session_id`
- `step_id`
- `proposal_id`
- `controller_id`
- `screenshot_id`
- `timestamp`
- `recoverable`

### 7.5 核心数据结构建议

为避免前后端理解不一致，建议先定义统一提议结构：

```ts
type ExploreProposal = {
  session_id: string;
  step_id: string;
  proposal_id: string;
  screenshot_id: string;
  proposal_kind: "act" | "wait" | "assert" | "branch" | "finish" | "ask_human";
  confidence: number;
  risk_level: "low" | "medium" | "high";
  reasoning_summary: string;
  expected_outcome: string;
  needs_human_attention: boolean;
  overlays: Array<{
    type: "roi" | "target" | "begin" | "end" | "hint" | "verify";
    rect: [number, number, number, number];
    label?: string;
    score?: number;
  }>;
  verification_plan: {
    mode: "visual_match" | "ocr_match" | "state_assert" | "manual_confirm";
    params: Record<string, unknown>;
  };
  node_draft: {
    label: string;
    recognition: {
      type: string;
      param: Record<string, unknown>;
    };
    action: {
      type: string;
      param: Record<string, unknown>;
    };
    others: Record<string, unknown>;
  };
  alternatives?: Array<{
    title: string;
    summary: string;
    expected_outcome: string;
    node_draft: ExploreProposal["node_draft"];
  }>;
};
```

同时建议补充一个会话级状态结构：

```ts
type ExploreWorkingMemory = {
  goal: string;
  confirmed_chain: Array<Record<string, unknown>>;
  user_corrections: Array<Record<string, unknown>>;
  recent_failures: Array<Record<string, unknown>>;
  safety_flags: string[];
  template_hints: Array<Record<string, unknown>>;
};
```

其中 `node_draft` 应尽量与现有 Pipeline Node 的 `recognition / action / others` 结构保持一致，降低落地成本。

### 7.6 Ghost Node 的工程落地策略

建议采用以下策略：

- **不新增持久化节点类型**，保持正式节点仍为现有 Pipeline Node
- 前端单独维护 `draftMeta`，如 `isGhost`、`proposalId`、`confidence`、`riskLevel`、`overlayBindings`
- 只有在用户确认后，才把 `node_draft` 写入现有流程 Store
- 未确认的 Ghost Node 不参与正常保存与导出

这样既能复用现有节点渲染与字段编辑能力，也能降低对当前画布模型的侵入性。

### 7.7 持久化、日志与回放

建议从一开始就为可恢复性预留结构：

- **会话快照**：保存最近一次会话状态、目标、步骤号、工作记忆和已确认链
- **截图缓存**：为最近若干步保留前后截图、标注信息和验证结果
- **审阅日志**：记录用户对每个提议的确认、修改、拒绝、回退和接管
- **Agent Trace**：记录 Observe / Plan / Review / Execute / Verify / Replan 的关键事件
- **回放能力**：后续可基于日志回放探索过程，用于调试、产品改进和模板沉淀

MVP 可以先做“最近会话恢复”，完整回放可放后续阶段。

## 8. 与现有功能的结合

### 8.1 LocalBridge 与 mfw 能力复用

- 复用现有截图、OCR、点击、滑动、输入等能力
- 探索模式不重新发明执行层，只新增会话、工作记忆和决策编排层
- 执行结果继续沿用既有协议风格，降低集成成本

### 8.2 与现有调试会话风格的结合

- Explore 会话可以复用 `DebugSessionV2` 的设计思路：后端维护会话，前端订阅状态与事件
- 协议风格建议沿用当前 `path + data` 模式，保持实现一致性
- 部分截图、执行回流和暂停 / 恢复能力可与现有调试体系共享实现

### 8.3 字段面板结合

- 当前 Ghost Node 可直接复用字段面板进行参数微调
- 用户修改后的字段应回写到当前提议，并覆盖 Agent 原始输出
- 现有智能补全能力继续作为字段级辅助存在

### 8.4 模板系统结合

- Agent 提议可优先参考已有模板和历史节点模式
- 用户确认且验证通过的高质量步骤可沉淀为模板
- 后续可实现“基于模板的探索偏置”，降低重复性场景的决策成本

### 8.5 现有 Store 与画布逻辑结合

建议新增独立的 `Explore Store`，而不是把探索状态硬塞进现有 Flow Store：

- **Explore Store**：会话状态、Agent 状态、提议、截图、标注、面板 UI、恢复信息
- **Flow Store**：正式节点、边、历史、视图、路径等既有编辑状态

两者的交界点只有一个：

> 当用户确认当前提议时，才把 `node_draft` 写入 Flow Store，并纳入正常历史记录。

## 9. 验收标准与指标

### 9.1 功能验收

MVP 至少应满足以下能力闭环：

1. 已连接控制器后，可以成功创建探索会话
2. 可以基于当前截图得到结构化提议、截图标注和验证计划
3. 用户可以确认、修改、拒绝、回退、接管、终止当前步骤
4. 确认后可以执行该步骤，并获得执行结果、验证结果与新截图
5. 验证失败时，系统能进入重规划或要求人工介入，而不是继续盲跑
6. 失败、断连、截图异常时，系统能给出清晰反馈并支持恢复或终止

### 9.2 体验指标

建议关注以下指标，用于衡量该模式是否真正有价值：

- **首个可运行流程完成时间**
- **单步提议确认率**
- **单步人工修改率**
- **验证失败后重规划成功率**
- **因执行失败触发回退的比例**
- **从截图到提议返回的耗时**
- **会话中断后的恢复成功率**

### 9.3 实施里程碑建议

| 里程碑 | 内容                                                    |
| ------ | ------------------------------------------------------- |
| **M1** | 定义 Explore 会话模型、工作记忆结构、协议和前端状态结构 |
| **M2** | 打通单步提议展示、Ghost Node、截图标注和审阅交互        |
| **M3** | 打通测试 / 执行 / 验证 / 重规划闭环                     |
| **M4** | 增强恢复能力、维护适配、模板沉淀与差异视图              |

## 10. 价值与愿景

### 10.1 用户价值

| 用户类型       | 获得价值                                |
| -------------- | --------------------------------------- |
| **新手**       | 降低学习门槛，在引导下完成首个 Pipeline |
| **熟练开发者** | 提升编写效率，减少机械重复劳动          |
| **维护者**     | 更快定位失效步骤，降低 UI 适配成本      |

### 10.2 长期愿景

- **把 AI 从补全器提升为协作 Agent**：它不只是帮用户填字段，而是帮用户推进任务、验证结果、处理偏差
- **个人知识沉淀**：用户的确认、修改与拒绝记录可形成个人偏好素材
- **团队知识共享**：高质量探索结果可沉淀为团队模板与最佳实践
- **生态共建**：未来可形成基于场景的探索策略库，而不是只沉淀零散节点

## 11. 相关概念区分

| 概念                | 说明                                           | 与本模式的关系                                                                 |
| ------------------- | ---------------------------------------------- | ------------------------------------------------------------------------------ |
| **提示词工程**      | 主要关注单次模型调用的输入组织与输出格式       | 本模式会使用 prompt，但 prompt 只是 Agent Runtime 的一个实现组件，不是整体架构 |
| **传统 AI 补全**    | 单节点级别的字段补全                           | 本模式是流程级协作，传统补全仍作为子能力存在                                   |
| **单 Agent 类架构** | 一个有状态、可调用工具、可吸收反馈的受约束代理 | 本模式当前阶段的推荐实现方式                                                   |
| **多 Agent 编排**   | 多个代理彼此分工协作                           | 当前阶段不是必需，待单 Agent 跑稳后再评估                                      |
| **MCP**             | 模型上下文协议，强调标准化工具调用             | 本模式当前不依赖 MCP，优先复用 LocalBridge 直接调用 mfw 的能力                 |
| **Skills / 模板**   | 文档、脚本或结构化经验的封装                   | 探索结果未来可进一步沉淀为模板、策略或 Skill                                   |
| **普通编辑模式**    | 用户直接在画布上手动编排节点                   | 探索模式是增强层，可随时切换回普通编辑模式                                     |

## 12. 待决策问题

以下问题建议在实现前进一步确认：

- MVP 默认接入哪类模型，是否优先支持本地模型
- Planner 与 Verifier 是否先由同一个模型承担，还是在后续阶段拆分
- 工作记忆是只保留内存态，还是需要落盘到本地缓存
- 截图进入模型前，是否需要增加 OCR / 候选框 / 页面摘要等预处理层
- “先测试”是否允许对真实设备造成轻量副作用，还是仅做纯识别验证
- 在维护适配场景中，是否需要专门的“旧节点 vs 新提议”对比视图
- 何时将“确认 + 验证通过”的步骤沉淀为模板候选
