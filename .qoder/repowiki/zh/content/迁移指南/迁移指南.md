## 简介
本指南面向从 YAMaaPE 迁移到 MaaPipelineEditor（MPE）的用户，系统讲解迁移流程、版本差异、兼容性处理、数据转换、协议兼容、配置持久化方案、常见问题与回滚策略，以及自动化迁移工具的最佳实践。文档基于仓库内现有实现进行归纳总结，帮助你安全、可控地完成迁移。

## 项目结构
MPE 前端以 React + React Flow 为核心，采用“解析器 + 编译器”的架构：导入时解析 Pipeline JSON，构建流程图；导出时将流程图编译为 Pipeline V2（默认）或 V1 格式。本地服务通过 WebSocket 协议与前端交互，支持文件列表、内容、变更通知、保存确认等。

## 核心组件
- 版本检测与标准化：自动识别节点的 recognition/action 字段版本，统一大小写与类型，保证兼容性。
- 导入器：解析 Pipeline JSON，迁移废弃字段（如 v5.1 的 interrupt/is_sub），构建节点、边与配置。
- 导出器：将流程图编译为 Pipeline JSON，支持 V1/V2 协议、分离/集成配置导出。
- 节点解析器：将 Flow 节点字段映射到 Pipeline 结构，处理识别/动作/其他参数及焦点字段。
- 配置解析器：识别并提取配置标记，兼容新旧版本标记字段。
- 字段索引：提供识别/动作/其他字段的 Schema 与键列表，支撑版本检测与参数匹配。
- 配置存储：集中管理导出行为（协议版本、是否导出默认识别/动作、配置处理模式等）。
- 协议层：文件/配置/调试协议通过 WebSocket 与本地服务通信，支持自动重载、保存确认等。

## 详细组件分析

### 从 YAMaaPE 迁移流程
- 旧项目无需整体适配 MPE，MPE 以单个 json 文件为操作单位；若旧文件无需改动，可直接导入使用。
- 仍保留 YAMaaPE 特殊字段兼容，但需将 MaaFramework 4.5 前的字段手动调整为 4.5 之后的字段。
- 使用 v1 协议导入后，MPE 会自动解析 YAMaaPE 的相关配置字段并适配为新字段，随后可用 v2 导入。

### 版本差异与兼容性处理
- 节点版本检测：识别 recognition/action 字段是否为字符串（V1）或对象（V2），并标准化类型大小写。
- 类型标准化：提供识别/动作类型的大小写归一化，若不在预定义集合则抛错，避免非法类型。
- 参数迁移：针对特定字段（如识别 method）进行数值迁移，确保与新协议一致。

### 废弃字段迁移（v5.1）
- interrupt 字段：迁移到 next，并统一添加 JumpBack 前缀或 jump_back 属性。
- is_sub 子节点：在 next/on_error 引用中自动添加 JumpBack 前缀或 jump_back 属性。
- 迁移完成后删除 is_sub 字段，保持结构简洁与规范。

### 配置持久化方案（集成 vs 分离）
- 集成方案：导出时将配置嵌入到 Pipeline 对象中，键名带有标记前缀，便于一次性保存。
- 分离方案：导出 Pipeline 与配置为两个独立 JSON，便于版本控制与团队协作。
- 配置处理模式：通过配置存储中的 configHandlingMode 控制导出行为，同时影响 isExportConfig 的同步。

```mermaid
flowchart TD

Mode --> |separated| Sep["导出分离配置"]
Mode --> |none| None["不导出配置"]
Int --> Out["输出完整对象"]
Sep --> Split["拆分 Pipeline 与配置"]
Split --> Out2["输出两份 JSON"]

```

## 性能考量
- 导入阶段：使用 JSONC 解析与键顺序遍历，避免不必要的深拷贝；仅在需要时执行自动布局。
- 导出阶段：按顺序排序节点，减少键顺序不确定性；分离导出时先合并再拆分，避免重复序列化。
- 协议层：保存确认采用超时机制，避免长时间阻塞；文件变更通知支持批量处理与自动重载。

## 故障排查指南
- 导入失败：检查 Pipeline 格式与版本一致性，查看控制台错误信息。
- 导出失败：若提示节点名重复，修改重复名称后重试。
- 保存失败：确认本地服务连接正常，查看保存确认回调状态。
- 文件变更：若未自动重载，可在设置中开启自动重载或手动重新加载。
