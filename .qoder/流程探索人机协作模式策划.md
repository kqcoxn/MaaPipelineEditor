# 流程探索人机协作模式

借鉴 RLHF（Reinforcement Learning from Human Feedback）中的"人在回路"思想，由 AI 给出初步方案，然后人调整验收后再进行下一轮。

---

## 一、交互流程

```
┌─────────────────────────────────────────────────────────────────┐
│                      流程探索人机协作模式                          │
├─────────────────────────────────────────────────────────────────┤
│  1. 用户输入总目标（自然语言描述）                                  │
│          ↓                                                       │
│  2. 截取当前画面 → AI 分析并给出方案 + 理由说明                      │
│          ↓                                                       │
│  3. 以 Ghost Node 形式展示方案（虚线边框预览态）                     │
│          ↓                                                       │
│  4. 用户操作：执行 / 确认 / 编辑 / 退出 / 完成                      │
│          ↓                                                       │
│  5. 确认后回到步骤 2 循环，直到手动完成或 AI 判断到达目标             │
└─────────────────────────────────────────────────────────────────┘
```

### 步骤详解

#### 步骤 1：输入总目标

用户以自然语言描述任务目标，例如：

- "完成每日签到任务"
- "从主界面进入设置页面"
- "领取所有邮件奖励"

#### 步骤 2：截图 + AI 分析

- 触发方式：每次判断下一步前自动截取当前画面
- 复用：现有 AI 预测功能（`collectNodeContext` + `predictNodeConfig`）
- 输出：方案 + 理由说明（解释为什么选择该识别类型、动作类型、参数推断依据）

#### 步骤 3：方案展示

- 以 **Ghost Node**（幽灵节点）形式展示
  - 虚线边框预览态，向用户传递「点击退出时不会保存」的视觉提示
  - 实质是真实的 Pipeline 节点，支持点选、字段面板编辑
- 复用：现有字段面板进行参数编辑

#### 步骤 4：用户操作

| 操作     | 含义                     | 行为                                             |
| -------- | ------------------------ | ------------------------------------------------ |
| **执行** | 在设备上实际运行这个动作 | 执行后自动截图，进入下一轮分析                   |
| **确认** | 采纳方案，正式加入流程图 | 节点转为正式态（取消虚线），截图进入下一轮       |
| **编辑** | 修改参数后再确认         | 打开字段面板编辑，编辑后仍需点击「确认」才算采纳 |
| **退出** | 结束探索，不保存当前节点 | 删除当前 Ghost Node，退出探索模式                |
| **完成** | 结束探索，保留当前节点   | 保留当前节点，退出探索模式                       |

#### 步骤 5：循环终止

- 用户手动点击「完成」或「退出」
- AI 判断已到达目标状态（需在步骤 2 中检测目标是否达成）

## 二、UI 展示形式

### 2.1 入口：悬浮球（FAB）

- **位置**：右下角，在排版工具面板（`bottom: 20px; right: 20px`）上方
- **形式**：圆形悬浮按钮（Floating Action Button）
- **交互**：点击呼出流程中控面板

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│                          画布区域                            │
│                                                             │
│                                            ┌─────────────┐  │
│                                            │ 悬浮球 (FAB) │  │
│                                            │    ◉ 探索    │  │
│                                            └─────────────┘  │
│                                            ┌─────────────┐  │
│                                            │ 排版工具面板  │  │
│                                            └─────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 流程中控面板

- **类型**：Panel（非 Modal）
- **位置**：右下角，悬浮球展开后显示
- **尺寸**：紧凑，不占用过多画布空间

#### 状态一：未进入探索

显示目标输入框 + 起始节点选择：

```
┌──────────────────────────────┐
│ 🎯 输入目标...               │
│                            [→]│
├──────────────────────────────┤
│ 起始节点：（可选）            │
│ [ 选择已有节点...         ▼ ] │
└──────────────────────────────┘
```

- **目标输入框**：必填，描述任务目标
- **起始节点选择**：可选，指定从哪个已有节点开始探索
  - 默认为空，表示从当前状态开始
  - 选择后，AI 会基于该节点的上下文进行预测

#### 状态二：探索中

显示当前状态和统计信息：

```
┌──────────────────────────────┐
│ 🎯 完成每日签到任务           │
├──────────────────────────────┤
│ 状态：等待确认                │
│ 已完成：3 步                  │
│                              │
│ [退出]              [完成]   │
└──────────────────────────────┘
```

### 2.3 Ghost Node 操作按钮

- **位置**：嵌入流程图，跟随幽灵节点
- **按钮**：仅「执行」「确认」两个
- **编辑**：点击节点即可，无需单独按钮
- **退出/完成**：在中控面板中

```
┌────────────────────────┐
│     Ghost Node         │
│    （虚线边框）          │
│                        │
│  [执行]    [确认]       │
└────────────────────────┘
```

### 2.4 避让规则

| 元素         | 位置                        | z-index |
| ------------ | --------------------------- | ------- |
| 悬浮球       | `bottom: 80px; right: 20px` | 20      |
| 中控面板     | 悬浮球上方展开              | 19      |
| 排版工具面板 | `bottom: 20px; right: 20px` | 10      |

## 四、与现有 MPE 功能结合

### 4.1 进入条件

探索模式需要设备连接才能正常工作：

| 检查项   | 未满足时行为                    |
| -------- | ------------------------------- |
| 设备连接 | 阻止进入，提示"请先连接设备"    |
| AI 配置  | 阻止进入，提示"请先配置 AI API" |

### 4.2 AI 预测复用

- **视觉模型**：直接使用 `sendVision` 发送截图，不依赖 OCR
- **一次预测**：每步只给一次 AI 预测结果
- **手动调整**：用户在字段面板自行调整参数或使用工具箱辅助

### 4.3 历史记录

复用现有 AI 对话历史面板：

- 每一步的 AI 推理依据自动保存
- 用户可随时回看

### 4.4 分支处理

采用 **线性预测 + 提醒** 策略：

1. AI 始终给出一个线性预测节点
2. 若检测到可能的分支场景，在推理依据中提醒用户
3. 用户根据提醒手动添加分支节点

## 五、异常处理与边界场景

### 5.1 设备断开

探索过程中设备断开时：

1. 提示用户设备已断开
2. 暂停探索状态，等待重连
3. 重连后可继续

### 5.2 用户中断

用户主动离开探索（如切换文件、关闭页面）：

1. 弹出确认提示：「当前探索未完成，是否保存已确认的节点？」
2. 选择「保存」→ 保留已确认节点，退出探索
3. 选择「不保存」→ 删除所有探索产生的节点（包括已确认），退出探索
4. 选择「取消」→ 继续探索

### 5.3 不支持的场景

| 场景               | 处理方式                 |
| ------------------ | ------------------------ |
| 回退到某步重新探索 | 不支持，有问题需退出重来 |
| 暂停后继续         | 不支持，中断即退出       |

## 七、实现架构

### 7.1 分层设计

采用**状态机 + 工具函数**的分层架构，预留智能体扩展能力：

```
┌─────────────────────────────────────────────────────────────┐
│                      UI 层（React 组件）                      │
│         悬浮球、中控面板、Ghost Node、字段面板               │
├─────────────────────────────────────────────────────────────┤
│                   状态层（Zustand Store）                     │
│       exploreStore：状态机 + 探索上下文 + 节点列表           │
├─────────────────────────────────────────────────────────────┤
│                   业务层（核心逻辑）                          │
│     exploreController：预测、执行、确认、回滚等操作          │
├─────────────────────────────────────────────────────────────┤
│                   工具层（原子能力）                          │
│   screenshot | execute | createNode | confirmNode | ...     │
├─────────────────────────────────────────────────────────────┤
│                   AI 层（视觉预测）                           │
│         predictNodeConfig + MaaMCP Skills                   │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 状态机设计

```typescript
// 探索状态枚举
enum ExploreStatus {
  IDLE = "idle", // 未开始
  PREDICTING = "predicting", // AI 预测中
  WAITING_CONFIRM = "waiting_confirm", // 等待用户确认
  EXECUTING = "executing", // 执行动作中
  PAUSED = "paused", // 设备断开暂停
  COMPLETED = "completed", // 已完成
}

// 探索上下文
interface ExploreContext {
  goal: string; // 用户目标
  startNodeId?: string; // 起始节点
  confirmedNodeIds: string[]; // 已确认节点
  ghostNodeId?: string; // 当前 Ghost Node
  stepCount: number; // 步骤计数
  status: ExploreStatus; // 当前状态
  lastPrediction?: AIPrediction; // 最近一次预测结果
}
```

### 7.3 核心操作

```typescript
// exploreController 提供的操作
const exploreController = {
  // 开始探索
  start: (goal: string, startNodeId?: string) => Promise<void>,

  // 执行当前方案
  execute: () => Promise<void>,

  // 确认当前方案
  confirm: () => Promise<void>,

  // 完成探索
  complete: () => void,

  // 退出探索
  abort: (saveConfirmed: boolean) => void,

  // 暂停（设备断开时）
  pause: () => void,

  // 恢复（设备重连后）
  resume: () => Promise<void>,
};
```

### 7.4 扩展能力预留

| 扩展方向        | 预留方式                                         |
| --------------- | ------------------------------------------------ |
| **AI 自主模式** | 状态机支持自动流转，只需添加「自动确认」决策逻辑 |
| **多目标并行**  | exploreStore 设计为数组，支持多个 ExploreContext |
| **回退重试**    | confirmedNodeIds 支持回溯，预留 undo 接口        |
| **智能体封装**  | 工具层独立，可直接包装为 Agent Tools             |

## 八、总结

流程探索人机协作模式的核心特点：

1. **人在回路**：AI 给方案，人确认，渐进式构建流程
2. **视觉驱动**：基于截图的视觉模型预测，不依赖 OCR
3. **最小化新增 UI**：复用现有字段面板、工具箱、AI 对话历史
4. **线性预测**：即使存在分支可能，也给出线性方案 + 提醒
